Angular 中的无障碍功能

edit

Web 会被各种各样的人使用，包括有视觉或运动障碍的人。有多种辅助技术能使这些人更轻松地和基于 Web 的软件应用进行交互。另外，将应用设计得更易于访问通常也能改善所有用户的体验。

有关设计无障碍应用问题和技术的深入介绍，请参阅 Google 的 web.dev Learn Accessibility 课程。

本页讨论了设计 Angular 应用的最佳实践，这些实践对所有用户（包括依赖辅助技术的用户）都适用。

无障碍属性（Attribute）

构建无障碍的 Web 体验通常需要设置 无障碍富互联网应用（ARIA）属性，以提供可能缺失的语义意义。 使用属性绑定模板语法来控制与无障碍相关的属性值。

在 Angular 中绑定 ARIA 属性（Attribute）时，必须使用 attr. 前缀。ARIA 规范针对的是 HTML 属性（Attribute），而不是 DOM 元素的属性（Property）。

      
<!-- Use attr. when binding to an ARIA attribute -->
<button [attr.aria-label]="myActionLabel">…</button>

注意： 这种语法仅对属性 绑定是必须的。 静态 ARIA 属性不需要额外语法。

      
<!-- Static ARIA attributes require no extra syntax -->
<button aria-label="Save document">…</button>

有帮助的： 按约定，HTML 属性使用小写名称（ tabindex），而属性使用 camelCase 名称（ tabIndex）。

有关属性和属性（Property）之间差异的更多背景信息，请参阅 绑定语法指南。

Angular UI 组件

由 Angular 团队维护的 Angular Material库是一套可复用的 UI 组件，旨在完全实现无障碍功能。 组件开发工具包（CDK）包括 a11y包，提供了支持各种无障碍功能的工具。例如：

  • LiveAnnouncer 用于通过 aria-live 区域为屏幕阅读器用户播报消息。有关更多信息，请参阅 W3C 文档 aria-live regions。

  • cdkTrapFocus指令会将 Tab 键的焦点限制在某个元素内。使用它可以为模态对话框等组件创建无障碍体验，确保焦点必须被限制。

有关这些和其他工具的详细信息，请参阅 Angular CDK 无障碍性概述。

增强原生元素

原生 HTML 元素捕获了一些对无障碍性很重要的标准交互模式。在制作 Angular 组件时，应尽可能直接复用这些原生元素，而不是重新实现已获良好支持的行为。

比如，你可以创建一个组件，它使用属性（Attribute）选择器指向原生 <button> 元素，而不是为各种新按钮创建自定义元素。通常这适用于 <button> 和 <a>，但也可以用于许多其它类型的元素。

你可以在 Angular Material 中看到这种模式的示例： MatButton、 MatTabNav和 MatTable。

将容器用于原生元素

有时要使用的原生元素需要一个容器元素。比如，原生 <input> 元素不能有子元素，因此任何自定义的文本输入组件都需要用额外的元素来包装 <input>。如果只在自定义组件的模板中包含 <input>，该组件的将用户无法为 input 元素设置任意 Property 和 Attribute。相反，你应该创建一个使用内容投影的容器组件，以将原生控件包含在组件的 API 中。

你可以将 MatFormField作为这种模式的示例。

路由

导航后的焦点管理

在 UI 中跟踪和控制 焦点是无障碍设计中的一个重要考虑因素。在使用 Angular 路由时，你应该决定导航后页面的焦点位置。

为了避免仅依赖视觉提示，你需要确保在页面导航后路由代码更新焦点。使用 Router服务的 NavigationEnd事件来知道何时更新焦点。

以下范例显示了导航后如何在 DOM 中查找并把焦点移动到主体内容的头部。

      
router.events.pipe(filter(e => e instanceof NavigationEnd)).subscribe(() => {
  const mainHeader = document.querySelector('#main-content-header')
  if (mainHeader) {
    mainHeader.focus();
  }
});

在实际的应用程序中，哪些元素会获得焦点取决于该应用特有的结构和布局。获得焦点的元素应使用户能够立即移动到刚刚进入视野的主要内容。你应该避免当路由变化后焦点重新回到 body 元素的情况。

活跃链接识别

应用于活跃的 RouterLink元素的 CSS 类，例如 RouterLinkActive，提供了一个视觉提示来识别活跃链接。不幸的是，视觉提示对盲人或视力障碍用户没有帮助。将 aria-current属性应用于元素可以帮助识别活跃链接。有关更多信息，请参阅 Mozilla 开发者网络（MDN）aria-current。

RouterLinkActive指令提供了 ariaCurrentWhenActive输入，当链接变为活跃状态时，设置 aria-current为指定值。

以下示例展示了如何将 active-page 类应用于活跃链接，以及如何在它们处于活跃状态时将它们的 aria-current 属性设置为 "page"：

      
    <nav>
      <a routerLink="home"
         routerLinkActive="active-page"
         ariaCurrentWhenActive="page">
        Home
      </a>
      <a routerLink="about"
         routerLinkActive="active-page"
         ariaCurrentWhenActive="page">
        About
      </a>
      <a routerLink="shop"
         routerLinkActive="active-page"
         ariaCurrentWhenActive="page">
        Shop
      </a>
    </nav>
