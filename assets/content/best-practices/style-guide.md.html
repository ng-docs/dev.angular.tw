Angular 編碼風格指南

edit

正在尋找關於 Angular 語法、約定和應用結構的權威指南？ 請看這裡。 本風格指南介紹了推薦的約定，更重要的是，解釋了原因。

風格詞彙表

每條準則都描述了一個好的或壞的實踐，並且都具有一致的呈現方式。

每條準則的措辭都表明了建議的強度。

Do 是指應該始終遵循的準則。 Always 可能有點太絕對了。 字面上應該始終遵循的準則非常罕見。 另一方面，你需要一個非常特殊的情況才能打破 Do 準則。

Consider 準則通常應該遵循。 如果你完全理解準則背後的含義，並且有充分的理由偏離，那麼就這樣做。 目標是保持一致。

Avoid 表示你應該幾乎從不做的某些事情。 avoid 的程式碼範例具有明顯的紅色標頭。

Why？
給出遵循先前建議的原因。

檔案結構約定

一些程式碼範例顯示了一個檔案，該檔案具有一個或多個類似命名的配套檔案。 例如，hero.component.ts 和 hero.component.html。

該指南使用快捷方式 hero.component.ts|html|css|spec 來表示這些不同的檔案。 使用此快捷方式使本指南的檔案結構更易於閱讀且更簡潔。

單一職責

將 單一職責原則 (SRP) 應用於所有元件、服務和其他符號。 這有助於使應用更清晰、更易於閱讀和維護，以及更易於測試。

單一規則

風格 01-01

Do 每個檔案定義一個事物，例如服務或元件。

Consider 將檔案限制為 400 行程式碼。

Why？
每個檔案一個元件使程式碼更易於閱讀、維護，並避免與原始碼控制中的團隊發生衝突。

Why？
每個檔案一個元件可以避免隱藏的錯誤，這些錯誤通常在將元件組合在一個檔案中時出現，在這些檔案中，元件可能會共享變數、建立不必要的閉包或與依賴項產生不必要的耦合。

Why？
單個元件可以是其檔案的預設匯出，這有助於透過路由器進行延遲載入。

關鍵是使程式碼更可複用、更易於閱讀且不易出錯。

以下反面範例在同一檔案中定義了 AppComponent、啟動了應用、 定義了 Hero 模型物件，並從伺服器載入了英雄。 不要這樣做。

命名

命名約定對於可維護性和可讀性至關重要。 本指南推薦了檔案名和符號名稱的命名約定。

通用命名準則

風格 02-01

Do 對所有符號使用一致的名稱。

Do 遵循一種模式，該模式描述符號的屬性，然後是其型別。 推薦的模式是 feature.type.ts。

Why？
命名約定有助於提供一種一致的方式來快速查詢內容。 專案內的一致性至關重要。 與團隊保持一致性很重要。 整個公司的一致性提供了巨大的效率。

Why？
命名約定應有助於更快地找到所需的程式碼，並使其更易於理解。

Why？
資料夾和檔案的名稱應清楚地表達其意圖。 例如，app/heroes/hero-list.component.ts 可能包含一個管理英雄清單的元件。

用點和破折號分隔檔案名

風格 02-02

Do 使用破折號分隔描述性名稱中的單詞。

Do 使用點分隔描述性名稱和型別。

Do 對所有元件使用一致的型別名稱，遵循一種模式，該模式描述元件的屬性，然後是其型別。 推薦的模式是 feature.type.ts。

Do 使用傳統的型別名稱，包括 .service、.component、.pipe、.module 和 .directive。 如果必須，可以建立其他型別名稱，但要注意不要建立太多。

Why？
型別名稱提供了一種一致的方式來快速識別檔案中的內容。

Why？
型別名稱使使用編輯器或 IDE 的模糊搜尋技術輕鬆找到特定檔案型別。

Why？
未縮寫的型別名稱（如 .service）具有描述性且明確。 諸如 .srv、.svc 和 .serv 之類的縮寫可能會造成混淆。

Why？
型別名稱為任何自動化任務提供模式對應。

符號和檔案名

風格 02-03

Do 對所有資產使用一致的名稱，這些名稱以它們所代表的內容命名。

Do 對類別名稱使用大駝峰命名法。

Do 將符號的名稱與檔案的名稱對應。

Do 為該類別型的事物在符號名稱後附加常規字尾（例如 Component、Directive、Module、Pipe 或 Service）。

要 為該類別型的檔案名新增約定俗成的字尾（例如 .component.ts、.directive.ts、.module.ts、.pipe.ts 或 .service.ts）。

為什麼？
一致的約定可以輕鬆快速地識別和引用不同型別的資產。

服務名

風格 02-04

要 對所有以屬性命名的服務使用一致的名稱。

要 在服務類別名稱後新增 Service 字尾。 例如，用於獲取資料或英雄的服務應該稱為 DataService 或 HeroService。

一些術語明確地表示服務。 它們通常以 "-er" 結尾來表示執行者。 你可能更喜歡將記錄訊息的服務命名為 Logger 而不是 LoggerService。 請確定你的專案中是否可以接受此例外。 與往常一樣，力求保持一致。

為什麼？
提供了一種一致的方式來快速識別和引用服務。

為什麼？
像 Logger 這樣清晰的服務名稱不需要字尾。

為什麼？
像 Credit 這樣的服務名稱是名詞，需要字尾，並且當不清楚它是服務還是其他東西時，應該使用字尾命名。

引導啟動

風格 02-05

要 將應用的引導啟動和平台邏輯放在名為 main.ts 的檔案中。

要 在引導啟動邏輯中包含錯誤處理。

避免 將應用邏輯放在 main.ts 中。 相反，請考慮將其放在元件或服務中。

為什麼？
遵循了應用啟動邏輯的一致約定。

為什麼？
遵循了其他技術平台中熟悉的約定。

指令選擇器

風格 02-06

應該 對指令的選擇器命名使用小駝峰命名法。

為什麼？
保持指令中定義的、繫結到檢視的屬性（Property）的名稱與屬性（Attribute）名稱一致。

為什麼？
Angular HTML 解析器是區分大小寫的，並且可以識別小駝峰命名法。

風格 02-08

為什麼？
防止名稱衝突。

為什麼？

應用結構和 NgModules

對實現有一個近期規劃，並有一個長期願景。 從小處著手，但要牢記應用的發展方向。

應用的所有程式碼都放在名為 src 的資料夾中。 所有屬性區域都在它們自己的資料夾中。

所有內容都是每個檔案一個資產。 每個元件、服務和通道都在其自己的檔案中。 所有第三方供應商指令碼都儲存在另一個資料夾中，而不是 src 資料夾中。 在本指南中使用檔案的命名約定。

總體結構指南

風格 04-06

應該 從小處著手，但要牢記應用未來的發展方向。

應該 對實現有一個近期規劃，並有一個長期願景。

應該 將應用的所有程式碼都放在名為 src 的資料夾中。

可以考慮 在元件有多個配套檔案（.ts、.html、.css 和 .spec）時，為其建立一個資料夾。

為什麼？
有助於在早期階段保持應用結構小巧且易於維護，同時在應用增長時易於演變。

為什麼？
元件通常有四個檔案（例如，*.html、*.css、*.ts 和 *.spec.ts），並且會很快使資料夾變得雜亂。

按屬性分資料夾 結構

風格 04-07

應該 建立以其代表的屬性區域命名的資料夾。

為什麼？
開發人員可以快速定位程式碼並識別每個檔案代表的內容。 該結構儘可能扁平，並且沒有重複或冗餘的名稱。

為什麼？
透過組織內容，幫助減少應用變得雜亂。

為什麼？
當檔案很多時，例如 10 個以上，使用一致的資料夾結構可以更容易地定位它們，而在扁平結構中則更困難。

欲知詳情，請參考此資料夾和檔案結構範例。

應用根模組

重要提示： 以下樣式指南建議適用於基於 NgModule 的應用。新的應用應該改用獨立的元件、指令和通道。

樣式 04-08

要在應用的根資料夾中建立一個 NgModule，例如，如果建立基於 NgModule 的應用，則在 /src/app 中建立。

為什麼？
每個基於 NgModule 的應用都至少需要一個根 NgModule。

考慮將根模組命名為 app.module.ts。

為什麼？
使其更容易定位和識別根模組。

app/app.module.ts

      
import {NgModule} from '@angular/core';import {BrowserModule} from '@angular/platform-browser';import {RouterModule} from '@angular/router';import {AppComponent} from './app.component';import {HeroesComponent} from './heroes/heroes.component';@NgModule({  imports: [    BrowserModule,    RouterModule.forChild([{path: '04-08', component: AppComponent}]),  ],  declarations: [AppComponent, HeroesComponent],  exports: [AppComponent],})export class AppModule {}

屬性模組

樣式 04-09

要為應用中所有不同的屬性建立一個 NgModule；例如，Heroes 屬性。

要將屬性模組放在與屬性區域同名的資料夾中；例如，在 app/heroes 中。

要命名屬性模組檔案，使其反映屬性區域和資料夾的名稱；例如，app/heroes/heroes.module.ts。

要命名屬性模組的符號，使其反映屬性區域、資料夾和檔案的名稱；例如，app/heroes/heroes.module.ts 定義了 HeroesModule。

為什麼？
屬性模組可以向其他模組公開或隱藏其實現。

為什麼？
屬性模組標識了構成屬性區域的不同相關元件集。

為什麼？
屬性模組可以很容易地被急性載入和延遲載入路由。

為什麼？
屬性模組在特定功能和其他應用屬性之間定義了清晰的邊界。

為什麼？
屬性模組有助於明確責任，並使不同團隊更容易分配開發職責。

為什麼？
屬性模組可以很容易地被隔離以進行測試。

共享屬性模組

樣式 04-10

要在 shared 資料夾中建立一個名為 SharedModule 的屬性模組；例如，app/shared/shared.module.ts 定義了 SharedModule。

要在一個共享模組中宣告元件、指令和通道，當這些條目將被其他屬性模組中宣告的元件複用和引用時。

考慮當共享模組的內容在整個應用中被引用時，使用名稱 SharedModule。

考慮不要在共享模組中提供服務。 服務通常是單例，它們為整個應用或特定的屬性模組提供一次。 但是，也有例外情況。 例如，在下面的範例程式碼中，請注意 SharedModule 提供了 FilterTextService。 這在這裡是可以接受的，因為該服務是無狀態的；也就是說，服務的消費者不受新實例的影響。

要匯入 SharedModule 中的資產所需的所有模組；例如，CommonModule 和 FormsModule。

為什麼？
SharedModule 將包含元件、指令和通道，它們可能需要來自另一個通用模組的屬性；例如，CommonModule 中的 ngFor。

要在 SharedModule 中宣告所有元件、指令和通道。

要從 SharedModule 匯出其他屬性模組需要使用的所有符號。

為什麼？
SharedModule 的存在是為了使常用的元件、指令和通道可用於許多其他模組中元件的樣板。

避免在 SharedModule 中指定應用範圍內的單例提供者。 有意的單例是可以的。 請注意。

為什麼？
延遲載入的屬性模組如果匯入了該共享模組，將建立該服務自己的副本，並可能產生不良結果。

為什麼？
你不希望每個模組都有其自己單獨的單例服務實例。 然而，如果 SharedModule 提供服務，則確實存在發生這種情況的風險。

HostListener/HostBinding 裝飾器與 host 元資料

風格 06-03

考慮優先使用 @HostListener 和 @HostBinding，而不是 @Directive 和 @Component 裝飾器的 host 屬性（Property）。

務必在選擇時保持一致。

為什麼？
與 @HostBinding 關聯的屬性（Property）或與 @HostListener 關聯的方法只能在一個地方修改，即指令的類別中。 如果你使用 host 元資料屬性（Property），則必須同時修改指令類中的屬性（Property）/方法宣告以及與該指令關聯的裝飾器中的元資料。

app/shared/validator.directive.ts

      
import {Directive, HostBinding, HostListener} from '@angular/core';@Directive({  selector: '[tohValidator]',})export class ValidatorDirective {  @HostBinding('attr.role') role = 'button';  @HostListener('mouseenter') onMouseEnter() {    // do work  }}

與不太推薦的 host 元資料替代方案進行比較。

為什麼？
host 元資料只有一個術語需要記住，並且不需要額外的 ES 匯入。

app/shared/validator2.directive.ts

      
import {Directive} from '@angular/core';@Directive({  selector: '[tohValidator2]',  host: {    '[attr.role]': 'role',    '(mouseenter)': 'onMouseEnter()',  },})export class Validator2Directive {  role = 'button';  onMouseEnter() {    // do work  }}

服務

服務是單例

風格 07-01

務必在同一個注入器中將服務用作單例。 使用它們來共享資料和功能。

為什麼？
服務非常適合在屬性區或應用中共享方法。

為什麼？
服務非常適合共享有狀態的記憶體資料。

import {inject, Injectable} from '@angular/core';import {HttpClient} from '@angular/common/http';import {Hero} from './hero.model';@Injectable()export class HeroService {  private http = inject(HttpClient);  getHeroes() {    return this.http.get<Hero[]>('api/heroes');  }}

提供服務

風格 07-03

務必在服務的 @Injectable 裝飾器中使用應用的根注入器來提供服務。

為什麼？
Angular 注入器是分層的。

為什麼？
當你向根注入器提供服務時，該服務的實例將被共享，並且在每個需要該服務的類別中都可用。 當服務共享方法或狀態時，這是理想的方案。

為什麼？
當你在服務的 @Injectable 裝飾器中註冊服務時，最佳化工具（例如 Angular CLI 的生產建立所使用的工具）可以執行搖樹最佳化並移除應用未使用的服務。

為什麼？
當兩個不同的元件需要服務的不同實例時，這不理想。 在這種情況下，最好在需要新的和單獨實例的元件級別提供服務。

src/app/treeshaking/service.ts

      
import {Injectable} from '@angular/core';@Injectable({  providedIn: 'root',})export class Service {}

使用 @Injectable() 類別裝飾器

風格 07-04

務必在使用型別作為服務的依賴的令牌時，使用 @Injectable() 類別裝飾器而不是 @Inject 引數裝飾器。

為什麼？
Angular 依賴注入（DI）機制基於服務建構式函式引數的宣告型別來求解服務自身的依賴。

為什麼？
當服務僅接受與型別令牌關聯的依賴項時，與在每個單獨的建構式函式引數上使用 @Inject() 相比，@Injectable() 語法要簡潔得多。

資料服務

透過服務與伺服器通訊

風格 08-01

務必將用於執行資料操作和與資料互動的邏輯重構到服務中。

務必使資料服務負責 XHR 呼叫、本地儲存、記憶體儲存或任何其他資料操作。

為什麼？
元件的職責是呈現和收集檢視的訊息。 它不應該關心如何獲取資料，而只需知道向誰請求資料即可。 分離資料服務會將關於如何獲取資料的邏輯轉移到資料服務，並使元件更簡單，更專注於檢視。

為什麼？
在測試使用資料服務的元件時，這使得更容易測試（模擬或真實）資料呼叫。

為什麼？
資料管理的細節（例如標頭、HTTP 方法、快取、錯誤處理和重試邏輯）與元件和其他資料使用者無關。

資料服務封裝了這些細節。 在服務內部演化這些細節更容易，而不會影響其使用者。 並且使用模擬服務實現來測試使用者也更容易。
