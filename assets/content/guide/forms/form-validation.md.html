驗證表單輸入

edit

你可以透過驗證使用者輸入的準確性和完整性來提高總體資料質量。 本頁介紹如何在回應式表單和樣板驅動表單中，驗證來自 UI 的使用者輸入並顯示有用的驗證訊息。

在樣板驅動表單中驗證輸入

要向樣板驅動表單新增驗證，你可以新增與 原生 HTML 表單驗證 相同的驗證 Attribute。 Angular 使用指令將這些 Attribute 與框架中的驗證器函式進行對應。

每次表單控制元件的值發生變化時，Angular 都會執行驗證，並生成驗證錯誤清單（導致 INVALID 狀態）或 null（導致 VALID 狀態）。

然後，你可以透過將 ngModel 匯出到本地樣板變數來檢查控制元件的狀態。 以下範例將 NgModel 匯出到一個名為 name 的變數中：

請注意該範例闡明的以下屬性。

  • <input> 元素帶有 HTML 驗證 Attribute：required 和 minlength。 它還帶有一個自定義驗證器指令 forbiddenName。 有關更多訊息，請參閱自定義驗證器部分。

  • #name="ngModel" 將 NgModel 匯出到一個名為 name 的本地變數中。 NgModel 映象了其底層 FormControl 實例的許多屬性（Property），因此你可以在樣板中使用它來檢查控制元件狀態，例如 valid 和 dirty。 有關控制元件屬性（Property）的完整清單，請參閱 AbstractControl API 參考。

    • <div> 元素上的 *ngIf 顯示一組巢狀的訊息 div，但前提是 name 無效且控制元件為 dirty 或 touched 狀態。

    • 每個巢狀的 <div> 都可以為可能的驗證錯誤之一顯示自定義訊息。 有針對 required、minlength 和 forbiddenName 的訊息。

提示： 為了防止驗證器在使用者有機會編輯表單之前顯示錯誤，你應該檢查控制元件的 dirty 或 touched 狀態。

  • 當用戶更改被監視欄位中的值時，控制元件將被標記為 『dirty』
  • 當用戶模糊表單控制元件元素時，控制元件將被標記為 『touched』

在回應式表單中驗證輸入

在回應式表單中，事實之源是元件類別。 你無需透過樣板中的 Attribute 新增驗證器，而是將驗證器函式直接新增到元件類別中的表單控制元件模型。 然後，每當控制元件的值發生更改時，Angular 都會呼叫這些函式。

驗證器函式

驗證器函式可以是同步的，也可以是非同步的。

驗證器型別 詳情
同步驗證器 同步函式，它接受一個控制元件實例並立即回傳一組驗證錯誤或 null。當你實例化 FormControl 時，將它們作為第二個引數傳入。
非同步驗證器 非同步函式，它接受一個控制元件實例並回傳一個 Promise 或可觀察者，該 Promise 或可觀察者稍後會發出 一組驗證錯誤或 null。當你實例化 FormControl 時，將它們作為第三個引數傳入。

出於效能原因，僅當所有同步驗證器都透過時，Angular 才會執行非同步驗證器。 每個驗證器都必須在設定錯誤之前完成。

內建驗證器函式

你可以選擇編寫自己的驗證器函式，也可以使用 Angular 的一些內建驗證器。

在樣板驅動表單中作為 Attribute 可用的相同內建驗證器（例如 required 和 minlength）都可以用作 Validators 類別中的函式。 有關內建驗證器的完整清單，請參閱 Validators API 參考。

要將演員表單更新為回應式表單，請使用一些相同的內建驗證器——這次是以函式形式，如下例所示。

reactive/actor-form-reactive.component.ts (驗證器函式)

      
import {Component} from '@angular/core';import {FormControl, FormGroup, Validators, ReactiveFormsModule} from '@angular/forms';import {forbiddenNameValidator} from '../shared/forbidden-name.directive';@Component({  selector: 'app-actor-form-reactive',  templateUrl: './actor-form-reactive.component.html',  styleUrls: ['./actor-form-reactive.component.css'],  imports: [ReactiveFormsModule],})export class HeroFormReactiveComponent {  skills = ['Method Acting', 'Singing', 'Dancing', 'Swordfighting'];  actor = {name: 'Tom Cruise', role: 'Romeo', skill: this.skills[3]};  actorForm: FormGroup = new FormGroup({    name: new FormControl(this.actor.name, [      Validators.required,      Validators.minLength(4),      forbiddenNameValidator(/bob/i), // <-- Here's how you pass in the custom validator.    ]),    role: new FormControl(this.actor.role),    skill: new FormControl(this.actor.skill, Validators.required),  });  get name() {    return this.actorForm.get('name');  }  get skill() {    return this.actorForm.get('skill');  }}

在此範例中，name 控制元件設定了兩個內建驗證器——Validators.required 和 Validators.minLength(4)——以及一個自定義驗證器 forbiddenNameValidator。

所有這些驗證器都是同步的，因此它們作為第二個引數傳遞。 請注意，你可以透過將函式作為陣列傳入來支援多個驗證器。

此範例還新增了一些 getter 方法。 在回應式表單中，你始終可以透過父組上的 get 方法訪問任何表單控制元件，但有時定義 getter 作為樣板的簡寫很有用。

如果再次檢視 name 輸入的樣板，它與樣板驅動的範例非常相似。

定義自定義驗證器

內建驗證器並不總是與你的應用的精確用例相對應，因此有時你需要建立一個自定義驗證器。

考慮一下上一個範例中的 forbiddenNameValidator 函式。 以下是該函式的定義。

shared/forbidden-name.directive.ts (forbiddenNameValidator)

      
import {Directive, input} from '@angular/core';import {  AbstractControl,  NG_VALIDATORS,  ValidationErrors,  Validator,  ValidatorFn,} from '@angular/forms';/** An actor's name can't match the given regular expression */export function forbiddenNameValidator(nameRe: RegExp): ValidatorFn {  return (control: AbstractControl): ValidationErrors | null => {    const forbidden = nameRe.test(control.value);    return forbidden ? {forbiddenName: {value: control.value}} : null;  };}@Directive({  selector: '[appForbiddenName]',  providers: [{provide: NG_VALIDATORS, useExisting: ForbiddenValidatorDirective, multi: true}],})export class ForbiddenValidatorDirective implements Validator {  forbiddenName = input<string>('', {alias: 'appForbiddenName'});  validate(control: AbstractControl): ValidationErrors | null {    return this.forbiddenName      ? forbiddenNameValidator(new RegExp(this.forbiddenName(), 'i'))(control)      : null;  }}

該函式是一個工廠，它接受一個正則表示式來檢測特定的停用名稱，並回傳一個驗證器函式。

在此範例中，停用的名稱是 『bob』，因此驗證器會拒絕任何包含 『bob』 的演員姓名。 在其他地方，它可能會拒絕 『alice』 或設定正則表示式對應的任何名稱。

forbiddenNameValidator 工廠回傳設定的驗證器函式。 該函式接受一個 Angular 控制元件物件，如果控制元件值有效，則回傳 null，否則回傳一個驗證錯誤物件。 驗證錯誤物件通常具有一個屬性（Property），其名稱是驗證鍵 'forbiddenName'，其值是可以插入到錯誤訊息中的任意值字典 {name}。

自定義非同步驗證器與同步驗證器類似，但它們必須回傳 Promise 或可觀察者，後者稍後會發出 null 或驗證錯誤物件。 對於可觀察者，可觀察者必須完成，此時表單將使用為驗證發出的最後一個值。

向樣板驅動表單新增自定義驗證器

在樣板驅動表單中，向樣板新增一個指令，其中該指令包裝了驗證器函式。 例如，相應的 ForbiddenValidatorDirective 充當 forbiddenNameValidator 的套件裝器。

Angular 識別指令在驗證過程中的作用，因為該指令使用 NG_VALIDATORS 提供者註冊了自身，如下例所示。 NG_VALIDATORS 是一個預定義的提供者，它具有可擴充套件的驗證器集合。

shared/forbidden-name.directive.ts (提供者)

      
import {Directive, input} from '@angular/core';import {  AbstractControl,  NG_VALIDATORS,  ValidationErrors,  Validator,  ValidatorFn,} from '@angular/forms';/** An actor's name can't match the given regular expression */export function forbiddenNameValidator(nameRe: RegExp): ValidatorFn {  return (control: AbstractControl): ValidationErrors | null => {    const forbidden = nameRe.test(control.value);    return forbidden ? {forbiddenName: {value: control.value}} : null;  };}@Directive({  selector: '[appForbiddenName]',  providers: [{provide: NG_VALIDATORS, useExisting: ForbiddenValidatorDirective, multi: true}],})export class ForbiddenValidatorDirective implements Validator {  forbiddenName = input<string>('', {alias: 'appForbiddenName'});  validate(control: AbstractControl): ValidationErrors | null {    return this.forbiddenName      ? forbiddenNameValidator(new RegExp(this.forbiddenName(), 'i'))(control)      : null;  }}

然後，指令類實現 Validator 介面，以便它可以輕鬆地與 Angular 表單整合。 以下是指令的其餘部分，以幫助你瞭解它是如何組合在一起的。

控制元件狀態 CSS 類

Angular 會自動將許多控制元件屬性映象到表單控制元件元素作為 CSS 類。 使用這些類可以根據表單的狀態設定表單控制元件元素的樣式。 目前支援以下類。

  • .ng-valid
  • .ng-invalid
  • .ng-pending
  • .ng-pristine
  • .ng-dirty
  • .ng-untouched
  • .ng-touched
  • .ng-submitted (僅限外圍表單元素)

在以下範例中，演員表單使用 .ng-valid 和 .ng-invalid 類來設定每個表單控制元件的邊框顏色。

跨欄位驗證

跨欄位驗證器是一個自定義驗證器，它比較表單中不同欄位的值，並接受或拒絕它們的組合。 例如，你可能有一個表單提供互不相容的選項，因此使用者可以選擇 A 或 B，但不能同時選擇兩者。 某些欄位值也可能依賴於其他欄位值；僅當也選擇了 A 時，才允許使用者選擇 B。

以下跨驗證範例展示了如何執行以下操作：

  • 根據兩個同級控制元件的值，驗證回應式或基於樣板的表單輸入，
  • 在使用者與表單互動且驗證失敗後，顯示描述性錯誤訊息。

這些範例使用跨驗證來確保演員在填寫『演員表單』時不會在其角色中重複使用相同的名字。 驗證器透過檢查演員姓名和角色是否不對應來實現這一點。

向回應式表單新增跨驗證

該表單具有以下結構：

      
const actorForm = new FormGroup({  'name': new FormControl(),  'role': new FormControl(),  'skill': new FormControl()});

請注意，name 和 role 是同級控制元件。 要在單個自定義驗證器中評估這兩個控制元件，你必須在公共祖先控制元件中執行驗證：FormGroup。 你查詢 FormGroup 的子控制元件，以便可以比較它們的值。

建立非同步驗證器

非同步驗證器實現 AsyncValidatorFn 和 AsyncValidator 介面。 這些介面與它們的同步對應項非常相似，但有以下區別。

  • validate() 函式必須回傳 Promise 或可觀察者，
  • 回傳的可觀察者必須是有限的，這意味著它必須在某個時候完成。 要將無限可觀察者轉換為有限可觀察者，請透過諸如 first、last、take 或 takeUntil 之類的過濾運算子通道化該可觀察者。

非同步驗證發生在同步驗證之後，並且僅在同步驗證成功時執行。 如果更基本的驗證方法已經發現無效輸入，則此檢查使表單可以避免潛在的昂貴非同步驗證過程（例如 HTTP 請求）。

非同步驗證開始後，表單控制元件進入 pending 狀態。 檢查控制元件的 pending 屬性，並使用它來提供有關正在進行的驗證操作的視覺反饋。

常見的 UI 模式是在執行非同步驗證時顯示載入動畫。 以下範例展示了如何在樣板驅動表單中實現此目的。

      
<input [(ngModel)]="name" #model="ngModel" appSomeAsyncValidator><app-spinner *ngIf="model.pending"></app-spinner>

實現自定義非同步驗證器

在以下範例中，非同步驗證器確保為演員分配的角色尚未被佔用。 新演員不斷試鏡，老演員不斷退休，因此無法提前檢索可用角色清單。 為了驗證潛在的角色條目，驗證器必須啟動非同步操作以查詢當前所有已分配角色的中央資料庫。

以下程式碼建立了驗證器類 UniqueRoleValidator，它實現了 AsyncValidator 介面。

      
import {Directive, forwardRef, inject, Injectable} from '@angular/core';import {  AsyncValidator,  AbstractControl,  NG_ASYNC_VALIDATORS,  ValidationErrors,} from '@angular/forms';import {catchError, map} from 'rxjs/operators';import {ActorsService} from './actors.service';import {Observable, of} from 'rxjs';@Injectable({providedIn: 'root'})export class UniqueRoleValidator implements AsyncValidator {  private readonly actorsService = inject(ActorsService);  validate(control: AbstractControl): Observable<ValidationErrors | null> {    return this.actorsService.isRoleTaken(control.value).pipe(      map((isTaken) => (isTaken ? {uniqueRole: true} : null)),      catchError(() => of(null)),    );  }}@Directive({  selector: '[appUniqueRole]',  providers: [    {      provide: NG_ASYNC_VALIDATORS,      useExisting: forwardRef(() => UniqueRoleValidatorDirective),      multi: true,    },  ],})export class UniqueRoleValidatorDirective implements AsyncValidator {  private readonly validator = inject(UniqueRoleValidator);  validate(control: AbstractControl): Observable<ValidationErrors | null> {    return this.validator.validate(control);  }}

actorsService 屬性使用 ActorsService 令牌的實例進行初始化，該令牌定義了以下介面。

      
interface ActorsService {  isRoleTaken: (role: string) => Observable<boolean>;}

在實際應用中，ActorsService 將負責向演員資料庫發出 HTTP 請求，以檢查角色是否可用。 從驗證器的角度來看，服務的實際實現並不重要，因此該範例可以僅針對 ActorsService 介面進行編碼。

當驗證開始時，UnambiguousRoleValidator 使用當前控制元件值委託給 ActorsService 的 isRoleTaken() 方法。 此時，控制元件被標記為 pending 並保持此狀態，直到從 validate() 方法回傳的可觀察者鏈完成。

isRoleTaken() 方法排程一個 HTTP 請求，該請求檢查角色是否可用，並回傳 Observable<boolean> 作為結果。 validate() 方法透過 map 運算子通道化回應，並將其轉換為驗證結果。

然後，該方法像任何驗證器一樣，如果表單有效，則回傳 null；如果表單無效，則回傳 ValidationErrors。 此驗證器使用 catchError 運算子處理任何潛在的錯誤。 在本例中，驗證器將 isRoleTaken() 錯誤視為成功的驗證，因為未能發出驗證請求並不一定意味著角色無效。 你可以以不同的方式處理錯誤，而是回傳 ValidationError 物件。

一段時間過後，可觀察者鏈完成，非同步驗證完成。 pending 標誌設定為 false，並且表單有效性已更新。

最佳化非同步驗證器的效能

預設情況下，所有驗證器都會在每次表單值更改後執行。 對於同步驗證器，這通常不會對應用效能產生明顯影響。 然而，非同步驗證器通常會執行某種 HTTP 請求來驗證控制元件。 每次擊鍵後都發送 HTTP 請求可能會給後端 API 帶來壓力，應儘可能避免。

你可以透過將 updateOn 屬性從 change （預設值）更改為 submit 或 blur 來延遲更新表單的有效性。

對於樣板驅動表單，在樣板中設定該屬性。

      
<input [(ngModel)]="name" [ngModelOptions]="{updateOn: 'blur'}">

對於回應式表單，在 FormControl 實例中設定該屬性。

      
new FormControl('', {updateOn: 'blur'});

與原生 HTML 表單驗證的互動

預設情況下，Angular 透過在封閉的 <form> 上新增 novalidate 屬性來停用 原生 HTML 表單驗證，並使用指令將這些屬性與框架中的驗證器函式進行對應。 如果你想結合使用原生驗證和基於 Angular 的驗證，你可以使用 ngNativeValidate 指令重新啟用它。 有關詳細訊息，請參閱 API 文件。
