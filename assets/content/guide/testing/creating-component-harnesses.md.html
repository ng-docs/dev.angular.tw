為你的元件建立測試工具

edit

開始之前

提示： 本指南假定你已閱讀過元件測試工具概覽指南。如果你是初次使用元件測試工具，請先閱讀該指南。

什麼時候建立測試工具才有意義？

Angular 團隊建議為在多處使用且具有一定使用者互動性的共享元件建立元件測試工具。這最常用於小部件函式庫和類似的複用元件。測試工具在這些情況下很有價值，因為它們為這些共享元件的使用者提供了一個良好支援的 API，以便與元件進行互動。使用測試工具的測試可以避免依賴於這些共享元件不可靠的實現細節，例如 DOM 結構和特定的事件監聽器。

對於僅在一個地方出現的元件（例如應用中的頁面），測試工具沒有提供太多好處。在這些情況下，元件的測試可以合理地依賴於此元件的實現細節，因為測試和元件是同時更新的。但是，如果你想在單元測試和端到端測試中都使用測試工具，那麼測試工具仍然提供一些價值。

CDK 安裝

元件開發工具包 (CDK) 是一組用於建立元件的行為原語。要使用元件測試工具，請首先從 npm 安裝 @angular/cdk。你可以使用 Angular CLI 從終端執行此操作：

      
ng add @angular/cdk

擴充套件 ComponentHarness

抽象類別 ComponentHarness 是所有元件測試工具的基底類別。要建立自定義元件測試工具，請擴充套件 ComponentHarness 並實現靜態屬性 hostSelector。

hostSelector 屬性標識 DOM 中與此測試工具子類別對應的元素。在大多數情況下，hostSelector 應該與相應的 Component 或 Directive 的選擇器相同。例如，考慮一個簡單的彈出元件：

      
@Component({  selector: 'my-popup',  template: `    <button (click)="toggle()">{{triggerText()}}</button>    @if (isOpen()) {      <div class="my-popup-content"><ng-content></ng-content></div>    }  `})class MyPopup {  triggerText = input('');  isOpen = signal(false);  toggle() {    this.isOpen.update((value) => !value);  }}

在這種情況下，元件的最小測試工具如下所示：

      
class MyPopupHarness extends ComponentHarness {  static hostSelector = 'my-popup';}

雖然 ComponentHarness 子類別僅需要 hostSelector 屬性，但大多數測試工具還應該實現一個靜態 with 方法來生成 HarnessPredicate 實例。過濾測試工具 部分更詳細地介紹了這一點。

在元件的 DOM 中查詢元素

ComponentHarness 子類別的每個實例都代表相應元件的特定實例。你可以透過 ComponentHarness 基底類別中的 host() 方法訪問元件的宿主元素。

ComponentHarness 還提供了幾種在元件的 DOM 中定位元素的方法。這些方法是 locatorFor()、locatorForOptional() 和 locatorForAll()。這些方法建立一些用於查詢元素的函式，它們並不直接查詢元素。這種方法可以防止快取對過期元素的引用。例如，當 ngIf 隱藏然後顯示一個元素時，結果是一個新的 DOM 元素；使用函式可以確保測試始終引用 DOM 的當前狀態。

有關不同 locatorFor 方法的完整清單詳細訊息，請參閱 ComponentHarness API 參考頁。

例如，上面討論的 MyPopupHarness 範例可以提供如下方法來獲取觸發器和內容元素：

      
class MyPopupHarness extends ComponentHarness {  static hostSelector = 'my-popup';  /** Gets the trigger element */  getTriggerElement = this.locatorFor('button');  /** Gets the content element. */  getContentElement = this.locatorForOptional('.my-popup-content');}

使用 TestElement 實例

TestElement 是一個抽象概念，旨在跨不同的測試環境（單元測試、WebDriver 等）工作。當使用工具時，你應該透過這個介面執行所有 DOM 互動。其他訪問 DOM 元素的方式，例如 document.querySelector()，並非在所有測試環境中都有效。

TestElement 有許多與底層 DOM 互動的方法，例如 blur()、click()、getAttribute() 等。有關方法的完整清單，請參閱 TestElement API 參考頁。

除非是元件使用者直接定義的元素（例如元件的宿主元素），否則不要向測試工具使用者公開 TestElement 實例。為內部元素公開 TestElement 實例會導致使用者依賴於元件的內部 DOM 結構。

相反，應為最終使用者可能採取的特定操作或他們可能觀察到的特定狀態提供更窄範圍的方法。例如，前面部分中的 MyPopupHarness 可以提供諸如 toggle 和 isOpen 之類別的方法：

      
class MyPopupHarness extends ComponentHarness {  static hostSelector = 'my-popup';  protected getTriggerElement = this.locatorFor('button');  protected getContentElement = this.locatorForOptional('.my-popup-content');  /** Toggles the open state of the popup. */  async toggle() {    const trigger = await this.getTriggerElement();    return trigger.click();  }  /** Checks if the popup us open. */  async isOpen() {    const content = await this.getContentElement();    return !!content;  }}

為子元件載入測試工具

較大的元件通常由子元件組成。你也可以在元件的測試工具中反映這種結構。ComponentHarness 上的每個 locatorFor 方法都有一個備用簽名，可用於定位子測試工具而不是元素。

有關不同 locatorFor 方法的完整清單，請參閱 ComponentHarness API 參考頁。

例如，考慮使用上面的彈出視窗建立的選單：

      
@Directive({  selector: 'my-menu-item'})class MyMenuItem {}@Component({  selector: 'my-menu',  template: `    <my-popup>      <ng-content></ng-content>    </my-popup>  `})class MyMenu {  triggerText = input('');  @ContentChildren(MyMenuItem) items: QueryList<MyMenuItem>;}

MyMenu 的測試工具然後可以利用 MyPopup 和 MyMenuItem 的其他測試工具：

      
class MyMenuHarness extends ComponentHarness {  static hostSelector = 'my-menu';  protected getPopupHarness = this.locatorFor(MyPopupHarness);  /** Gets the currently shown menu items (empty list if menu is closed). */  getItems = this.locatorForAll(MyMenuItemHarness);  /** Toggles open state of the menu. */  async toggle() {    const popupHarness = await this.getPopupHarness();    return popupHarness.toggle();  }}class MyMenuItemHarness extends ComponentHarness {  static hostSelector = 'my-menu-item';}

使用 HarnessPredicate 過濾工具實例

當頁面包含特定元件的多個實例時，你可能希望根據元件的某些屬性進行過濾，以獲取特定的元件實例。例如，你可能想要具有某些特定文字的按鈕，或具有特定 ID 的選單。HarnessPredicate 類別可以為 ComponentHarness 子類別捕獲這樣的條件。雖然測試作者能夠手動構造 HarnessPredicate 實例，但當 ComponentHarness 子類別提供輔助方法來構造常用過濾器的謂詞時，會更容易。

你應該在每個 ComponentHarness 子類別上建立一個靜態 with() 方法，該方法回傳該類別的 HarnessPredicate。這允許測試作者編寫易於理解的程式碼，例如 loader.getHarness(MyMenuHarness.with({selector: '#menu1'}))。除了標準的選擇器和祖先選項之外，with 方法還應新增對特定子類別有意義的任何其他選項。

需要新增其他選項的測試工具應擴充套件 BaseHarnessFilters 介面和根據需要新增其他可選屬性。HarnessPredicate 提供了幾種用於新增選項的便捷方法：stringMatches()、addOption() 和 add()。有關完整說明，請參閱 HarnessPredicate API 頁面。

例如，當使用選單時，根據觸發器文字進行過濾以及根據選單條目的文字進行過濾非常有用：

      
interface MyMenuHarnessFilters extends BaseHarnessFilters {  /** Filters based on the trigger text for the menu. */  triggerText?: string | RegExp;}interface MyMenuItemHarnessFilters extends BaseHarnessFilters {  /** Filters based on the text of the menu item. */  text?: string | RegExp;}class MyMenuHarness extends ComponentHarness {  static hostSelector = 'my-menu';  /** Creates a `HarnessPredicate` used to locate a particular `MyMenuHarness`. */  static with(options: MyMenuHarnessFilters): HarnessPredicate<MyMenuHarness> {    return new HarnessPredicate(MyMenuHarness, options)        .addOption('trigger text', options.triggerText,            (harness, text) => HarnessPredicate.stringMatches(harness.getTriggerText(), text));  }  protected getPopupHarness = this.locatorFor(MyPopupHarness);  /** Gets the text of the menu trigger. */  async getTriggerText(): Promise<string> {    const popupHarness = await this.getPopupHarness();    return popupHarness.getTriggerText();  }  ...}class MyMenuItemHarness extends ComponentHarness {  static hostSelector = 'my-menu-item';  /** Creates a `HarnessPredicate` used to locate a particular `MyMenuItemHarness`. */  static with(options: MyMenuItemHarnessFilters): HarnessPredicate<MyMenuItemHarness> {    return new HarnessPredicate(MyMenuItemHarness, options)        .addOption('text', options.text,            (harness, text) => HarnessPredicate.stringMatches(harness.getText(), text));  }  /** Gets the text of the menu item. */  async getText(): Promise<string> {    const host = await this.host();    return host.text();  }}

你可以將 HarnessPredicate 而不是 ComponentHarness 類別傳遞給 HarnessLoader、LocatorFactory 或 ComponentHarness 上的任何 API。這允許測試作者在建立測試工具實例時輕鬆定位特定的元件實例。它還允許測試工具作者利用相同的 HarnessPredicate 在其測試工具類別上啟用更強大的 API。例如，考慮上面顯示的 MyMenuHarness 上的 getItems 方法。新增過濾 API 允許測試工具的使用者搜尋特定的選單條目：

      
class MyMenuHarness extends ComponentHarness {  static hostSelector = 'my-menu';  /** Gets a list of items in the menu, optionally filtered based on the given criteria. */  async getItems(filters: MyMenuItemHarnessFilters = {}): Promise<MyMenuItemHarness[]> {    const getFilteredItems = this.locatorForAll(MyMenuItemHarness.with(filters));    return getFilteredItems();  }  ...}

為使用內容投影的元素建立 HarnessLoader

某些元件將其他內容投影到元件的範本中。有關更多訊息，請參閱內容投影指南。

當你為使用內容投影的元件建立測試工具時，新增一個作用域限定為包含 <ng-content> 元素的 HarnessLoader 實例。這允許測試工具的使用者為作為內容傳入的任何元件載入其他測試工具。ComponentHarness 有幾種方法可用於為此類別情況建立 HarnessLoader 實例：harnessLoaderFor()、harnessLoaderForOptional()、harnessLoaderForAll()。有關更多詳細訊息，請參閱 HarnessLoader 介面 API 參考頁。

例如，上面範例中的 MyPopupHarness 可以擴充套件 ContentContainerComponentHarness，以新增對在元件的 <ng-content> 中載入測試工具的支援。

      
class MyPopupHarness extends ContentContainerComponentHarness<string> {  static hostSelector = 'my-popup';}

訪問元件宿主元素之外的元素

有時，元件測試工具可能需要訪問其對應元件的宿主元素之外的元素。例如，顯示浮動元素或彈出視窗的程式碼通常將 DOM 元素直接附著到文件主體，例如 Angular CDK 中的 Overlay 服務。

在這種情況下，ComponentHarness 提供了一種方法，可用於獲取文件根元素的 LocatorFactory。LocatorFactory 支援與 ComponentHarness 基底類別相同的大多數 API，然後可以用於相對於文件根元素進行查詢。

考慮一下，如果上面的 MyPopup 元件對彈出視窗內容使用了 CDK 浮層，而不是其自身範本中的元素。在這種情況下，MyPopupHarness 將不得不透過 documentRootLocatorFactory() 方法訪問內容元素，該方法獲取以文件根為根的定位器工廠。

      
class MyPopupHarness extends ComponentHarness {  static hostSelector = 'my-popup';  /** Gets a `HarnessLoader` whose root element is the popup's content element. */  async getHarnessLoaderForContent(): Promise<HarnessLoader> {    const rootLocator = this.documentRootLocatorFactory();    return rootLocator.harnessLoaderFor('my-popup-content');  }}

等待非同步任務

TestElement 上的方法會自動觸發 Angular 的變更檢測並等待 NgZone 內的任務。在大多數情況下，測試工具作者無需進行特殊操作即可等待非同步任務。但是，在某些極端情況下，這可能不足以滿足需求。

在某些情況下，Angular 動畫可能需要第二個變更檢測週期和隨後的 NgZone 穩定化，然後動畫事件才能完全重新整理。在需要這種情況時，ComponentHarness 提供了一個 forceStabilize() 方法，可以呼叫該方法來執行第二輪操作。

你可以使用 NgZone.runOutsideAngular() 在 NgZone 之外排程任務。如果需要顯式等待 NgZone 之外的任務，請在相應的測試工具上呼叫 waitForTasksOutsideAngular() 方法，因為這不會自動發生。
