元件測試基礎

edit

與 Angular 應用的所有其他部分不同，元件將 HTML 範本和 TypeScript 類別結合在一起。 元件實際上是範本和類別協同工作的產物。 為了充分地測試元件，你應該測試它們是否按預期協同工作。

此類別測試需要像 Angular 那樣在瀏覽器 DOM 中建立元件的宿主元素，並根據元件範本的描述，研究元件類別與 DOM 之間的互動。

正如你在後續章節中看到的那樣，Angular 的 TestBed 促進了這種測試。 但在許多情況下，僅測試元件類別本身，而無需 DOM 參與，就可以用更直接、更明顯的方式驗證元件的大部分行為。

元件 DOM 測試

元件不僅僅是它的類別。 元件還會與 DOM 和其他元件互動。 僅憑類別本身無法告訴你元件是否會正確渲染、回應使用者輸入和手勢，或者與其父元件和子元件整合。

  • Lightswitch.clicked() 是否繫結到任何東西，以便使用者可以呼叫它？
  • Lightswitch.message 是否顯示？
  • 使用者是否真的可以選擇 DashboardHeroComponent 顯示的英雄？
  • 英雄的名字是否按預期顯示（例如大寫）？
  • WelcomeComponent 範本中的歡迎訊息是否顯示？

對於前面闡明的簡單元件來說，這些可能不是令人困擾的問題。 但是，許多元件與其範本中描述的 DOM 元素具有複雜的互動，導致 HTML 隨著元件狀態的變化而出現和消失。

要回答這類別問題，你必須建立與元件關聯的 DOM 元素，必須檢查 DOM 以確認元件狀態在適當的時候正確顯示，並且必須模擬使用者與螢幕的互動，以確定這些互動是否會導致元件按預期執行。

要編寫這類別測試，你將使用 TestBed 的其他屬性以及其他測試輔助工具。

CLI 生成的測試

預設情況下，當你要求 CLI 生成新元件時，它會為你建立一個初始測試檔案。

例如，以下 CLI 指令在 app/banner 資料夾中生成一個 BannerComponent（帶有行內範本和樣式）：

      
ng generate component banner --inline-template --inline-style --module app

它還會為元件生成一個初始測試檔案 banner-external.component.spec.ts，如下所示：

此檔案中只有最後三行實際測試了元件，它們所做的只是斷言 Angular 可以建立該組件。

重要提示： 不要在呼叫 createComponent 後重新設定 TestBed。

nativeElement

ComponentFixture.nativeElement 的值具有 any 型別。 稍後你將遇到 DebugElement.nativeElement，它也具有 any 型別。

Angular 在編譯時無法知道 nativeElement 是哪種 HTML 元素，甚至無法知道它是否是 HTML 元素。 該應用可能正在非瀏覽器平台（例如伺服器或 Web Worker）上執行，在這些平台上，該元素可能具有縮減的 API，或者根本不存在。

本指南中的測試旨在在瀏覽器中執行，因此 nativeElement 值將始終是 HTMLElement 或其衍生類別之一。

知道它某種型別的 HTMLElement，請使用標準 HTML querySelector 更深入地探查元素樹。

這種迂迴的元素路徑是有充分理由的。

nativeElement 的屬性取決於執行時環境。 你可能正在非瀏覽器平台上執行這些測試，該平台沒有 DOM 或其 DOM 模擬不支援完整的 HTMLElement API。

Angular 依賴於 DebugElement 抽象，以便在所有受支援的平台上安全地工作。 Angular 沒有建立 HTML 元素樹，而是建立了一個 DebugElement 樹，該樹包裝了執行時平台的原生元素。 nativeElement 屬性解開 DebugElement 並回傳特定於平台的元素物件。

由於本指南的範例測試旨在僅在瀏覽器中執行，因此這些測試中的 nativeElement 始終是一個 HTMLElement，你可以在測試中探索其熟悉的方法和屬性。

By.css()

儘管本指南中的測試都在瀏覽器中執行，但有些應用可能至少在某些時候會在不同的平台上執行。

例如，作為一種讓應用在網路連線不良的裝置上啟動更快的策略，元件可能首先在服務器上渲染。伺服器端渲染器可能不支援完整的 HTML 元素 API。如果它不支援 querySelector，則之前的測試可能會失敗。

DebugElement 提供了適用於所有受支援平台的查詢方法。這些查詢方法接受一個謂詞函式，當 DebugElement 樹中的節點符合選擇標準時，該函式回傳 true。

你可以藉助從執行時平台函式庫中匯入的 By 類別來建立一個謂詞。以下是瀏覽器平台的 By 匯入：

一些值得注意的觀察結果：

  • By.css() 靜態方法使用標準 CSS 選擇器選擇 DebugElement 節點。
  • 該查詢回傳段落的 DebugElement。
  • 你必須解開該結果才能獲得段落元素。

當您透過 CSS 選擇器進行過濾，並且僅測試瀏覽器*原生元素*的屬性（Property）時，By.css 方法可能有點矯枉過正。

使用標準的 HTMLElement 方法（如 querySelector() 或 querySelectorAll()）進行過濾通常更直接和清晰。
