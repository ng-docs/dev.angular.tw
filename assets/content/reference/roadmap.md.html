Angular 路线图

edit

了解 Angular 团队如何在 Web 上积聚势头。

作为一个开源项目，Angular 的日常提交、PR 和势头都可以在 GitHub 上追踪。为了增加对日常工作如何与框架的未来联系的透明度，我们的路线图汇集了团队当前和未来的计划愿景。

以下项目不与特定的 Angular 版本相关联。我们将在完成后发布它们，并且它们将根据我们的发布计划成为特定版本的一部分，遵循语义化版本控制。例如，我们在完成后发布特性到下一个次要版本，或者如果它们包含破坏性更改，则发布到下一个主要版本。

目前，Angular 对框架有两个目标：

  1. 改进 Angular 开发者体验和
  2. 提升 框架的性能。

继续阅读，了解我们计划如何通过具体项目工作实现这些目标。

改进 Angular 开发者体验

开发者速度

交付 Angular Signals

此项目通过引入 Signals 作为响应式原语重新思考了 Angular 的响应式模型。初步规划产生了数百次讨论、与开发者的对话、反馈会、用户体验研究和一系列 RFC，收到了超过 1000 条评论。

作为 v17 版本的一部分，我们将 Angular Signals 库从开发者预览阶段提升出来。在 v18 中，你现在可以在开发者预览中使用基于 signal 的查询、输入和模型输入。接下来，我们将继续处理社区反馈，然后将这些 API 提升到稳定版本，并将 Signals 更深入地集成到 Angular 的变更检测机制中。

Zoneless 的 Angular

在 v18 中，我们在 Angular 中提供了实验性的 Zoneless 支持。这使得开发者可以在不包含 zone.js 的情况下使用该框架，从而提升性能、调试体验和互操作性。作为初始发布的一部分，我们还为 Angular CDK 和 Angular Material 引入了 Zoneless 支持。

下一步，我们将继续迭代 API 以改善开发者体验。

局部模板变量

局部模板变量是 Angular 问题追踪器中投票最多的特性之一。在 2024 年第二季度，我们开始了初步设计和原型开发。预计在 2024 年晚些时候进行更新。

改进 Angular Material 和 CDK

新的 CDK 原语

我们正在开发新的 CDK 原语，以便根据 WAI-ARIA 设计模式创建自定义组件，如 组合框。Angular v14 引入了稳定的 菜单和对话框原语 作为此项目的一部分，并在 v15 中引入了列表框。

Angular 组件无障碍性

我们正在根据 WCAG 等无障碍性标准评估 Angular Material 中的组件，并努力解决此过程中出现的任何问题。

改进工具

使用 ng test 现代化单元测试工具

在 v12 中，我们通过用 Cypress、Nightwatch 和 Webdriver.io 等现代替代工具替换 Protractor，重新审视了 Angular 端到端测试体验。接下来，我们想处理 ng test 以现代化 Angular 的单元测试体验。在 2023 年第二季度，我们引入了实验性的 Jest 支持，并 宣布从 Karma 过渡到 Web Test Runner。

今年晚些时候，我们将继续推进将 Web Test Runner 作为 Karma 替代品的进程。

使用 Language Service 简化独立导入

作为该举措的一部分，语言服务将自动在独立应用程序和基于 NgModule 的应用程序中导入组件和管道。此外，为了实现更小的应用包，我们将致力于让语言服务向你建议自动删除未使用的导入。

部分水合的探索

在 v17 中，我们将水合从开发者预览版中毕业，并且我们一直在观察到 40-50% 的 LCP 改进。自那时起，我们开始了部分水合的原型设计，并在 ng-conf 上展示了一个演示。预计在 2024 年提供实验性支持。

事件重播与 SSR 和预渲染

在 v18 中，我们引入了在使用服务器端渲染或预渲染时的事件重播功能。对于此特性，我们依赖于事件分发原语（以前称为 jsaction），该原语在 Google.com 上运行。

在接下来的几个月里，我们将收集社区对此特性的反馈，并努力使其成为稳定版。

改进 HMR（热模块重载）

Angular CLI 目前通过 ng serve --hmr 支持 HMR。在底层，这主要是从头开始重新渲染 Angular 应用，这比完全重新加载页面要好，但肯定可以改进。最重要的是，我们在此的策略应该是优化任何给定更改的周转时间，并与该类更改的频率相适应。未来，我们的团队将探索多个改进 HMR 的机会，包括：

  • 快速跟踪仅 CSS 的更改，并将其应用到页面上的任何现有组件。
  • 快速处理 Angular 模板专属的更改，并将其应用于页面上的任何现有组件。

通过统一的提交消息标准提高开发者一致性

我们希望统一Angular存储库（angular/angular, angular/components和angular/angular-cli）中的提交消息要求和一致性，以使我们的开发流程更加一致，并重复使用基础设施工具。

完成于 2021 年第 2 季度

将Angular语言服务过渡到Ivy

该项目的目标是通过将语言服务过渡到Ivy来提升体验并消除传统依赖。今天，语言服务仍然使用View Engine编译器和类型检查，即使对于Ivy应用程序也是如此。我们希望使用Ivy模板解析器和改进的类型检查来匹配应用程序行为的Angular语言服务。这一迁移也是解除View Engine限制的一步，这将简化Angular，减小npm包大小，并提高框架的可维护性。

完成于 2021 年第 2 季度

通过Angular中的本地Trusted Types提高安全性

与Google安全团队合作，我们正在添加对新Trusted Types API的支持。这个Web平台API帮助开发人员构建更安全的Web应用程序。

完成于 2021 年第 2 季度

通过Angular CLI webpack 5优化构建速度和捆绑包大小

作为v11版本的一部分，我们在Angular CLI中引入了webpack 5的可选择预览。为了确保稳定性，我们将继续改进实现，以实现构建速度和捆绑包大小的改进。

完成于 2021 年第 2 季度

通过在Universal应用程序中内联关键样式来加快应用程序

加载外部样式表是一个阻塞操作，这意味着浏览器在加载所有引用的CSS之前无法开始渲染应用程序。在页面的标头中有需要阻止渲染的资源可能会显著影响其加载性能，例如，首次内容绘制。为了加快应用程序，我们一直与Google Chrome团队合作，内联关键CSS并异步加载其余样式。

完成于 2021 年第 1 季度

通过更好的 Angular 错误消息改进调试

错误消息通常提供了有限的可操作信息，帮助开发人员解决问题。我们一直在努力通过添加相关代码、制定指南和其他材料来使错误消息更易发现，以确保更顺畅的调试体验。

完成于 2021 年第 1 季度

通过更新的入门文档改进开发者入职体验

我们将重新定义用户学习路径并刷新入门文档。我们将清晰地阐明 Angular 的优势，如何探索其能力，并提供指导，使开发人员尽快熟练掌握该框架。

完成于 2021 年第 1 季度

扩展组件测试工具最佳实践

Angular CDK 在版本 9 中向 Angular 引入了组件测试工具的概念。测试工具让组件作者为测试组件交互创建支持的 API。我们正在继续改进这种工具架构，并澄清使用测试工具的最佳实践。我们还致力于推动 Google 内部更多地采用这些测试工具。

完成于 2021 年第 1 季度

编写内容投影指南

内容投影是 Angular 的核心概念，在文档中并没有得到应有的关注。作为这个项目的一部分，我们希望确定内容投影的核心用例和概念，并加以记录。

完成于 2021 年第 2 季度

迁移到 ESLint

随着 TSLint 的废弃，我们将转向 ESLint。作为这一过程的一部分，我们将努力确保与当前推荐的 TSLint 配置向后兼容，为现有的 Angular 应用实施迁移策略，并在 Angular CLI 工具链中引入新的工具。

完成于 2020 年第 4 季度
