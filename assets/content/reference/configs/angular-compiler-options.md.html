Angular 編譯器選項

edit

當你使用預先編譯 (AoT) 時，你可以透過在TypeScript 設定檔案中指定 Angular 編譯器選項來控制應用的編譯方式。

Angular 選項物件 angularCompilerOptions 是 compilerOptions 物件的同級物件，後者為 TypeScript 編譯器提供標準選項。

tsconfig.json

      
/* To learn more about this file see: https://www.typescriptlang.org/docs/handbook/tsconfig-json.html. */{  "compileOnSave": false,  "compilerOptions": {    "baseUrl": "./",    "outDir": "./dist/out-tsc",    "forceConsistentCasingInFileNames": true,    "strict": true,    "noImplicitOverride": true,    "noPropertyAccessFromIndexSignature": true,    "noImplicitReturns": true,    "noFallthroughCasesInSwitch": true,    "sourceMap": true,    "declaration": false,    "downlevelIteration": true,    "experimentalDecorators": true,    "moduleResolution": "node",    "importHelpers": true,    "target": "es2020",    "module": "es2020",    "lib": [      "es2020",      "dom"    ]  },  "angularCompilerOptions": {    "enableI18nLegacyMessageIdFormat": false,    "strictInjectionParameters": true,    "strictInputAccessModifiers": true,    "strictTemplates": true  }}

使用 extends 進行設定繼承

與 TypeScript 編譯器類似，Angular AoT 編譯器也支援 TypeScript 設定檔案的 angularCompilerOptions 部分中的 extends。 extends 屬性位於頂層，與 compilerOptions 和 angularCompilerOptions 並列。

TypeScript 設定可以使用 extends 屬性從另一個檔案繼承設定。 來自基礎檔案的設定選項會先載入，然後被繼承設定檔案中的選項覆蓋。

例如：

tsconfig.app.json

      
/* To learn more about this file see: https://www.typescriptlang.org/docs/handbook/tsconfig-json.html. */{    "extends": "./tsconfig.json",    "compilerOptions": {      "outDir": "./out-tsc/app",      "types": []    },    "files": [      "src/main.ts"    ],    "include": [      "src/**/*.d.ts"    ],    "exclude": [      "src/test.ts",      "src/**/*.spec.ts",      "src/**/*-specs.ts",      "src/**/*.avoid.ts",      "src/**/*.0.ts",      "src/**/*.1.ts",      "src/**/*.1b.ts",      "src/**/*.2.ts",      "src/**/*.3.ts",      "src/**/*.4.ts",      "src/**/*.5.ts",      "src/**/*.6.ts",      "src/**/*.7.ts",      "src/**/testing"    ],    "angularCompilerOptions": {      "strictTemplates": true,      "preserveWhitespaces": true,      "sourceMap": true,      "declaration": false    }  }

更多訊息，請參閱《TypeScript 手冊》。

範本選項

以下選項可用於設定 Angular AoT 編譯器。

annotationsAs

修改發出 Angular 專用註解的方式以改進搖樹最佳化。 非 Angular 註解不受影響。 可以是 static fields 或 decorators 之一。預設值為 static fields。

  • 預設情況下，編譯器會將裝飾器替換為類別中的靜態欄位，這允許像 Closure compiler 這樣的高階搖樹最佳化器刪除未使用的類別

  • decorators 值將裝飾器保留在原位，這使得編譯速度更快。 TypeScript 發出對 __decorate 輔助函式的呼叫。 使用 --emitDecoratorMetadata 進行執行時反射。

    提示：生成的程式碼無法正確進行搖樹最佳化。

annotateForClosureCompiler

當為 true 時，使用 Tsickle 來使用 JSDoc 註解來註解發出的 JavaScript，Closure Compiler 需要這些註解。 預設為 false。

compilationMode

指定要使用的編譯模式。 以下模式可用：

模式 詳情
'full' 根據當前正在使用的 Angular 版本生成完全 AoT 編譯的程式碼。
'partial' 以穩定的中間形式生成程式碼，適用於已發布的函式庫。

預設值為 'full'。

對於大多數應用來說，'full' 是正確的編譯模式。

對於獨立發布的函式庫（例如 NPM 套件），請使用 'partial'。 'partial' 編譯輸出一種穩定的中間格式，這種格式可以更好地支持由以不同於函式庫的 Angular 版本建立的應用使用。 與應用一起在 『HEAD』 建立的函式庫，以及在單一程式碼儲存庫中使用相同 Angular 版本的函式庫可以使用 'full'，因為不存在版本偏差的風險。

disableExpressionLowering

當為 true（預設值）時，轉換註解中正在使用或可能正在使用的程式碼，以允許從模板工廠模組匯入它。 有關更多訊息，請參閱元資料重寫。

當為 false 時，停用此重寫，需要手動完成重寫。

disableTypeScriptVersionCheck

當為 true 時，編譯器不會檢視 TypeScript 版本，並且在使用不受支援的 TypeScript 版本時不會報告錯誤。 不建議這樣做，因為不受支援的 TypeScript 版本可能具有未定義的行為。 預設為 false。

enableI18nLegacyMessageIdFormat

指示 Angular 範本編譯器為透過 i18n 屬性在範本中標記的訊息建立舊式 ID。 有關標記訊息以進行本地化的更多訊息，請參閱標記要翻譯的文字。

除非你的專案依賴於先前使用舊式 ID 建立的翻譯，否則請將此選項設定為 false。 預設為 true。

pre-Ivy 訊息提取工具為提取的訊息 ID 建立了各種舊式格式。 這些訊息格式存在一些問題，例如空格處理以及依賴於範本原始 HTML 中的訊息。

新的訊息格式對空格更改更具彈性，在所有翻譯檔案格式中都相同，並且可以直接從對 $localize 的呼叫建立。 這允許應用程式碼中的 $localize 訊息使用與元件範本中相同的 i18n 訊息相同的 ID。

enableResourceInlining

當為 true 時，將所有 @Component 裝飾器中的 templateUrl 和 styleUrls 屬性替換為 template 和 styles 屬性中的行內內容。

啟用後，ngc 的 .js 輸出不包含任何延遲載入的範本或樣式 URL。

對於使用 Angular CLI 建立的函式庫專案，開發設定預設為 true。

enableLegacyTemplate

當為 true 時，啟用已棄用的 <template> 元素來代替 <ng-template>。 預設為 false。 某些第三方 Angular 函式庫可能需要此選項。

flatModuleId

用於匯入扁平模組的模組 ID（當 flatModuleOutFile 為 true 時）。 範本編譯器建立的引用在從扁平模組匯入符號時使用此模組名稱。 如果 flatModuleOutFile 為 false，則忽略此選項。

flatModuleOutFile

當為 true 時，生成給定檔案名的扁平模組索引和相應的扁平模組元資料。 用於建立與 @angular/core 和 @angular/common 類似打包的扁平模組。 當使用此選項時，函式庫的 package.json 應該引用建立的扁平模組索引，而不是函式庫索引檔案。

僅生成一個 .metadata.json 檔案，其中包含從函式庫索引匯出的符號所需的所有元資料。 在建立的 .ngfactory.js 檔案中，扁平模組索引用於匯入符號。符號包括來自函式庫索引的公共 API 和隱藏的內部符號。

預設情況下，在 files 欄位中提供的 .ts 檔案被假定為函式庫索引。 如果指定了多個 .ts 檔案，則使用 libraryIndex 來選擇要使用的檔案。 如果提供了多個 .ts 檔案但沒有 libraryIndex，則會產生錯誤。

使用給定的 flatModuleOutFile 名稱在與函式庫索引 .d.ts 檔案相同的位置建立一個扁平模組索引 .d.ts 和 .js。

例如，如果一個函式庫使用 public_api.ts 檔案作為模組的函式庫索引，則 tsconfig.json files 欄位將為 ["public_api.ts"]。 然後可以將 flatModuleOutFile 選項設定為例如 "index.js"，這將生成 index.d.ts 和 index.metadata.json 檔案。 函式庫的 package.json 的 module 欄位將為 "index.js"，而 typings 欄位將為 "index.d.ts"。

fullTemplateTypeCheck

當為 true（推薦值）時，啟用範本編譯器的繫結表示式驗證階段。此階段使用 TypeScript 來驗證繫結表示式。 有關更多訊息，請參閱範本型別檢查。

預設為 false，但是當你使用 Angular CLI 指令 ng new --strict 時，它在新專案的設定中被設定為 true。

重要提示： fullTemplateTypeCheck 選項已在 Angular 13 中棄用，取而代之的是 strictTemplates 系列編譯器選項。

generateCodeForLibraries

當為 true 時，為具有相應 .metadata.json 檔案的 .d.ts 檔案建立工廠檔案（.ngfactory.js 和 .ngstyle.js）。預設值為 true。

當為 false 時，僅為 .ts 檔案建立工廠檔案。 當使用工廠摘要時，請執行此操作。

preserveWhitespaces

當為 false（預設值）時，從已編譯的範本中刪除空白文字節點，這將導致發出的模板工廠模組更小。 設定為 true 以保留空白文字節點。

提示： 當使用水合時，建議你使用 preserveWhitespaces: false，這是預設值。如果你選擇透過將 preserveWhitespaces: true 新增到你的 tsconfig 來啟用保留空格，則你可能會遇到水合問題。這還不是完全支援的設定。確保伺服器和用戶端 tsconfig 檔案之間也保持一致設定。有關更多詳細訊息，請參閱水合指南。

skipMetadataEmit

當為 true 時，不生成 .metadata.json 檔案。 預設為 false。

.metadata.json 檔案包含範本編譯器從 .ts 檔案中所需的訊息，這些訊息未包含在 TypeScript 編譯器生成的 .d.ts 檔案中。 例如，此訊息包括註解的內容，例如元件的範本，TypeScript 會將其發出到 .js 檔案，但不會發出到 .d.ts 檔案。

當使用工廠摘要時，你可以設定為 true，因為工廠摘要包含 .metadata.json 檔案中訊息的副本。

如果你正在使用 TypeScript 的 --outFile 選項，請設定為 true，因為元資料檔案對於這種樣式的 TypeScript 輸出無效。 Angular 社群不建議將 --outFile 與 Angular 一起使用。 請改用捆綁器，例如 webpack。

skipTemplateCodegen

當為 true 時，不會發出 .ngfactory.js 和 .ngstyle.js 檔案。 這將關閉大多數範本編譯器功能，並停用範本診斷訊息的報告。

可用於指示範本編譯器生成 .metadata.json 檔案，以便與 npm 套件一起分發。 這避免了生成無法分發到 npm 的 .ngfactory.js 和 .ngstyle.js 檔案。

對於使用 Angular CLI 建立的函式庫專案，開發設定預設為 true。

strictMetadataEmit

當為 true 時，如果 "skipMetadataEmit" 為 false，則向 .metadata.json 檔案報告錯誤。 預設為 false。 僅當 "skipMetadataEmit" 為 false 且 "skipTemplateCodegen" 為 true 時使用。

此選項旨在驗證為與 npm 套件捆綁而發出的 .metadata.json 檔案。 驗證非常嚴格，並且可能會針對元資料發出錯誤，而這些元資料在範本編譯器使用時永遠不會產生錯誤。 你可以選擇透過在記錄符號的註解中包含 @dynamic 來禁止此選項為匯出的符號發出的錯誤。

.metadata.json 檔案包含錯誤是有效的。 如果元資料用於確定註解的內容，則範本編譯器會報告這些錯誤。 元資料收集器無法預測哪些符號旨在用於註解中。 它會搶先在匯出的符號的元資料中包含錯誤節點。 然後，如果使用了這些符號，範本編譯器可以使用錯誤節點來報告錯誤。

如果函式庫的用戶端打算在註解中使用符號，則範本編譯器通常不會報告此錯誤。 它會在用戶端實際使用該符號後才報告。 此選項允許在函式庫的建立階段檢測這些錯誤，例如，在生成 Angular 函式庫本身時使用。

對於使用 Angular CLI 建立的函式庫專案，開發設定預設為 true。

strictInjectionParameters

當為 true 時，會為注入型別無法確定的提供的引數報告錯誤。 當為 false 時，對於帶有 @Injectable 標記且型別無法求解的類別的建構式函式引數，會產生警告。 建議值為 true，但預設值為 false。

當你使用 Angular CLI 指令 ng new --strict 時，它在已建立專案的設定中設定為 true。

strictTemplates

當為 true 時，啟用嚴格範本型別檢查。

此選項啟用的嚴格性標誌允許你開啟和關閉特定型別的嚴格範本型別檢查。 請參閱排查範本錯誤。

當你使用 Angular CLI 指令 ng new --strict 時，它在新專案的設定中設定為 true。

strictStandalone

當為 true 時，如果元件、指令或通道不是獨立的，則報告錯誤。

trace

當為 true 時，在編譯範本時列印額外訊息。 預設為 false。

指令行選項

大多數時候，你都是使用Angular CLI間接與 Angular 編譯器互動。 在除錯某些問題時，你可能會發現直接呼叫 Angular 編譯器很有用。 你可以使用 @angular/compiler-cli npm 套件提供的 ngc 指令從命令行呼叫編譯器。

ngc 指令是 TypeScript 的 tsc 編譯器指令的套件裝器。 Angular 編譯器主要透過 tsconfig.json 設定，而 Angular CLI 主要透過 angular.json 設定。

除了設定檔案之外，你還可以使用 tsc 指令行選項來設定 ngc。
