工作区和项目文件结构

edit

你可以在 Angular 工作区的上下文中开发应用。 一个工作区包含一个或多个项目的文件。 一个项目是一组包含应用或可共享库的文件。

Angular CLI 的 ng new 命令可以创建一个工作区。

      
ng new my-project

当你运行此命令时，CLI 会在一个新工作区中安装必要的 Angular npm 包和其他依赖项，并创建一个名为 my-project 的根级别应用。

默认情况下， ng new 会在工作区的根目录创建一个初始的骨架应用及其端到端测试。 骨架是一个简单的欢迎应用，准备运行且易于修改。 根级应用与工作区同名，源文件位于工作区的 src/ 子文件夹中。

这种默认行为适用于典型的“多重（multi）仓库”开发风格，每个应用都位于它自己的工作区中。建议初学者和中级用户使用 ng new 为每个应用创建一个单独的工作区。

Angular 还支持包含 多个项目 的工作区。 这种开发环境适合开发可共享库的高级用户，以及采用单一代码仓开发风格的企业，该风格使用单个代码仓和全局配置来管理所有 Angular 项目。

要建立一个单一代码仓工作区，你应该跳过创建根应用。 参见下面的 多项目工作区设置。

工作区配置文件

工作区内的所有项目共享一个 配置。 工作区的顶层包含工作区级别的配置文件、根级应用的配置文件，以及根级应用源文件和测试文件的子文件夹。

工作区配置文件 目的
.editorconfig 代码编辑器的配置。 参见 EditorConfig。
.gitignore 指定 Git 应忽略的有意不跟踪的文件。 参见 Git。
README.md 工作区文档。
angular.json 工作区中所有项目的 CLI 配置，包括如何构建、启动和测试每个项目的配置选项。详情请参见 Angular 工作区配置。
package.json 为工作区中所有项目配置 npm 包依赖。有关此文件的具体格式和内容，请参见 npm 文档。
package-lock.json 提供由 npm 客户端安装到 node_modules中的所有包的版本信息。详情请参见 npm 文档。
src/ 根级应用项目的源文件。
node_modules/ 整个工作区安装的 npm 包。工作区范围内的 node_modules依赖对所有项目可见。
tsconfig.json 工作区中项目的基础 TypeScript配置。所有其他配置文件都继承自此基础文件。更多信息请参见 相关 TypeScript 文档。

应用项目文件

CLI 命令 ng new my-app 会默认创建名为 “my-app” 的工作区文件夹，并在 src/ 文件夹下为工作区顶层的根应用生成一个新的应用骨架。新生成的应用包含一个根模块的源文件，包括一个根组件及其模板。

当工作区文件结构就位后，可以在命令行中使用 ng generate命令为应用添加功能和数据。该根级应用是 CLI 命令的 默认应用（除非在创建 其他应用后更改默认设置）。

对于单应用的工作区，工作区的 src/ 子文件夹包含根应用的源文件（应用逻辑、数据和静态资源）。对于多项目的工作区，projects/ 文件夹中的其它项目各自包含一个具有相同结构的 project-name/src/ 子目录。

应用源文件

src/顶级目录下的文件支持运行你的应用。子文件夹包含应用源代码和特定于应用的配置。

应用支持文件 目的
app/ 包含定义你的应用逻辑和数据的组件文件。详情请参见 下文。
public/ 包含图像和其他资产文件，由开发服务器作为静态文件提供，并在构建应用时按原样复制。
favicon.ico 在书签栏中使用的应用图标。
index.html 当有人访问你的网站时所提供的主 HTML 页面。CLI 在构建应用时会自动添加所有 JavaScript 和 CSS 文件，因此通常不需要在这里手动添加任何 <script>或 <link>标签。
main.ts 你的应用的主入口点。
styles.css 应用程序全局 CSS 样式。

在 src/ 文件夹里面，app/ 文件夹中包含此项目的逻辑和数据。Angular 组件、模板和样式也都在这里。

src/app/文件 目的
app.config.ts 定义应用配置，告诉 Angular 如何组装应用。当你向应用添加更多提供者时，应在此声明。

仅在使用--standalone选项时生成。
app.component.ts 定义应用的根组件，名为 AppComponent。与此根组件关联的视图成为视图层次结构的根，你添加组件和服务到应用时，此视图将扩展。
app.component.html 定义与 AppComponent 关联的 HTML 模板。
app.component.css 定义 AppComponent 的 CSS 样式表。
app.component.spec.ts 定义 AppComponent 的单元测试。
app.module.ts 定义根模块，名为 AppModule，告诉 Angular 如何组装应用。初始时仅声明 AppComponent。当你向应用添加更多组件时，必须在此声明它们。

仅在使用 --standalone false 选项时生成。
app.routes.ts 定义应用的路由配置。

应用配置文件

根应用的特定配置文件位于工作区根目录。对于多项目工作区，项目特定的配置文件位于项目根目录下的 projects/project-name/。

项目特定的 TypeScript 配置文件继承自工作区范围的 tsconfig.json。

应用特定的配置文件 目的
tsconfig.app.json 应用特定的 TypeScript 配置，包括 Angular 编译器选项。
tsconfig.spec.json 应用测试的 TypeScript 配置。

多重项目

多项目工作区适合使用单一代码仓和全局配置进行多个 Angular 项目（「单一代码仓」模型）的组织。多项目工作区还支持库开发。

建立多项目工作区

如果你打算在工作区中包含多个项目，可以在创建工作区时不要自动创建初始应用，并为工作区指定一个唯一的名字。下列命令用于创建一个包含全工作区级配置文件的工作区，但没有根应用。

      
ng new my-workspace --no-create-application

然后，你可以使用工作区内唯一的名字来生成应用和库。

      
cd my-workspace
ng generate application my-app
ng generate library my-lib

多重项目的文件结构

第一个显式生成的应用进入 projects 文件夹，与工作区中的所有其他项目一起。新生成的库也添加在 projects 下。当你以这种方式创建项目时，工作区的文件结构完全与 工作区配置文件 angular.json 的结构一致。

      
my-workspace/
  ├── …                (workspace-wide configuration files)
  └── projects/        (applications and libraries)
      ├── my-app/      (an explicitly generated application)
      │   └── …        (application-specific code and configuration)
      └── my-lib/      (a generated library)
          └── …        (library-specific code and configuration)

库项目文件

当你使用 CLI 生成库（如使用命令 ng generate library my-lib）时，生成的文件进入工作区的 projects/ 文件夹。有关创建自己的库的更多信息，请参阅 创建库。

与应用不同，库有自己的 package.json 配置文件。

在 projects/ 目录下，my-lib 文件夹中包含你的库代码。

库源文件 目的
src/lib 包含你的库项目的逻辑和数据。和应用项目一样，库项目可以包含组件、服务、模块、指令和管道。
src/public-api.ts 指定从你的库中导出的所有文件。
ng-package.json 由 ng-packagr 用于构建你的库的配置文件。
package.json 配置此库所需的 npm 包依赖。
tsconfig.lib.json 库专用的 TypeScript 配置，包括 Angular 编译器选项。
tsconfig.lib.prod.json 在生产模式下构建库时使用的库专用 TypeScript 配置。
tsconfig.spec.json 用于库的单元测试的 TypeScript 配置。
