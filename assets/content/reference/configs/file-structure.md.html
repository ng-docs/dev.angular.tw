工作區和專案檔案結構

edit

你可以在 Angular 工作區的上下文中開發應用。 一個工作區包含一個或多個專案的相關檔案。 一個專案是由構成應用或可共享庫的一組檔案。

Angular CLI 的 ng new 指令會建立一個工作區。

      
ng new my-project

當你執行此指令時，CLI 會在一個新的工作區中安裝必要的 Angular npm 套件和其他依賴項，以及一個名為 my-project 的根級應用。

預設情況下，ng new 會在工作區的根級別建立一個初始骨架應用，以及它的端到端測試。 此骨架是一個簡單的歡迎應用，可以立即執行並且易於修改。 根級應用與工作區同名，並且原始碼檔案位於工作區的 src/ 子目錄下。

這種預設行為適用於典型的『多程式碼倉庫』開發風格，即每個應用都位於其自己的工作區中。 建議初學者和中級使用者使用 ng new 為每個應用建立一個單獨的工作區。

Angular 還支援具有多個專案的工作區。 這種型別的開發環境適用於正在開發可共享庫的高階使用者，以及使用『單一程式碼倉庫』開發風格的企業，這種風格為所有 Angular 專案使用單一程式碼倉庫和全域設定。

要建立單一程式碼倉庫工作區，你應該跳過建立根應用。 請參閱下面的建立多專案工作區。

工作區設定檔案

工作區內的所有專案共享一個設定。 工作區的頂層包含工作區範圍的設定檔案、根級應用的設定檔案以及根級應用原始碼和測試檔案的子資料夾。

工作區設定檔案 用途
.editorconfig 程式碼編輯器的設定。請參閱 EditorConfig。
.gitignore 指定 Git 應該忽略的，有意不跟蹤的檔案。
README.md 工作區的文件。
angular.json 工作區中所有專案的 CLI 設定，包括如何建立、啟動伺服器和測試每個專案的設定選項。 有關詳細訊息，請參閱Angular 工作區設定。
package.json 設定可用於工作區中所有專案的npm 套件依賴。 有關此檔案的特定格式和內容，請參閱 npm 文件。
package-lock.json 為 npm 用戶端安裝到 node_modules 中的所有套件提供版本訊息。 有關詳細訊息，請參閱 npm 文件。
src/ 根級應用專案的源檔案。
public/ 包含影象和其他資產檔案，這些檔案將由開發伺服器作為靜態檔案啟動伺服器，並在你建立應用時按原樣複製。
node_modules/ 為整個工作區安裝的 npm 套件。 工作區範圍的 node_modules 依賴項對所有專案都可見。
tsconfig.json 工作區中專案的基本TypeScript 設定。 所有其他設定檔案都從此基本檔案繼承。 有關更多訊息，請參閱相關的 TypeScript 文件。

應用專案檔案

預設情況下，CLI 指令 ng new my-app 建立一個名為『my-app』的工作區資料夾，並在工作區頂層的 src/ 資料夾中生成一個新的應用骨架。 新生成的應用套件含根模組的源檔案，以及根元件和樣板。

當工作區檔案結構就位後，你可以在指令行上使用 ng generate 指令嚮應用新增功能和數據。 這個初始的根級應用是 CLI 指令的預設應用（除非你在建立其他應用後更改了預設值）。

對於單應用工作區，工作區的 src 子資料夾包含根應用的源檔案（應用邏輯、資料和資產）。 對於多專案工作區，projects 資料夾中的其他專案包含一個具有相同結構的 project-name/src/ 子資料夾。

應用源檔案

src/ 頂層的檔案支援執行你的應用。 子資料夾包含應用原始碼和應用特定的設定。

應用支援檔案 用途
app/ 包含在其中定義應用邏輯和數據的元件檔案。 有關詳細訊息，請參見下面。
favicon.ico 要在書籤欄中用於此應用的圖示。
index.html 當有人訪問你的站點時啟動伺服器的主 HTML 頁面。 CLI 在建立你的應用時會自動新增所有 JavaScript 和 CSS 檔案，因此你通常不需要在此處手動新增任何 <script> 或 <link> 標籤。
main.ts 應用的main入口點。
styles.css 應用於整個應用的全域 CSS 樣式。

在 src 資料夾中，app 資料夾包含你專案的邏輯和數據。 Angular 元件、樣板和樣式放在這裡。

src/app/ 檔案 用途
app.config.ts 定義應用設定，以告知 Angular 如何組裝應用。 當你嚮應用新增更多提供者時，應該在此處宣告它們。

僅在使用 --standalone 選項時生成。
app.component.ts 定義應用的根元件，名為 AppComponent。 當你嚮應用新增元件和服務時，與此根元件關聯的檢視將成為檢視分層的根。
app.component.html 定義與 AppComponent 關聯的 HTML 樣板。
app.component.css 定義 AppComponent 的 CSS 樣式表。
app.component.spec.ts 定義 AppComponent 的單元測試。
app.module.ts 定義根模組，名為 AppModule，它告訴 Angular 如何組裝應用。 最初僅宣告 AppComponent。 當你嚮應用新增更多元件時，必須在此處宣告它們。

僅在使用 --standalone false 選項時生成。
app.routes.ts 定義應用的路由設定。

應用設定檔案

根應用的特定於應用的設定檔案位於工作區根級別。 對於多專案工作區，特定於專案的設定檔案位於專案根目錄下的 projects/project-name/ 中。

特定於專案的 TypeScript 設定檔案繼承自工作區範圍的 tsconfig.json。

應用特定的設定檔案 用途
tsconfig.app.json 應用特定的TypeScript 設定，包括 Angular 編譯器選項。
tsconfig.spec.json 用於應用測試的 TypeScript 設定。

多個專案

多專案工作區適用於為多個 Angular 專案（『單一程式碼倉庫』模型）使用單一程式碼倉庫和全域設定的組織。 多專案工作區還支援庫開發。

建立多專案工作區

如果你打算在一個工作區中擁有多個專案，那麼可以在建立工作區時跳過初始應用的生成步驟，併為該工作區指定一個唯一的名稱。 以下指令會建立一個包含所有工作區範圍的設定檔案的工作區，但不包含根級別的應用。

      
ng new my-workspace --no-create-application

然後，你可以生成名稱在工作區內唯一的應用和函式庫。

      
cd my-workspaceng generate application my-appng generate library my-lib

多專案檔案結構

第一個顯式生成的應用會進入 projects 資料夾，與工作區中的所有其他專案放在一起。 新生成的函式庫也會新增到 projects 下。 當你以這種方式建立專案時，工作區的檔案結構與 工作區設定檔案 angular.json 的結構完全一致。

      
my-workspace/  ├── …                (workspace-wide configuration files)  └── projects/        (applications and libraries)      ├── my-app/      (an explicitly generated application)      │   └── …        (application-specific code and configuration)      └── my-lib/      (a generated library)          └── …        (library-specific code and configuration)

函式庫專案檔案

當你使用 CLI 生成函式庫（使用諸如 ng generate library my-lib 這樣的指令）時，生成的檔案會進入工作區的 projects/ 資料夾。 有關建立你自己的函式庫的更多訊息，請參閱 建立函式庫。

與應用不同，函式庫有自己的 package.json 設定檔案。

在 projects/ 資料夾下，my-lib 資料夾包含你的函式庫程式碼。

函式庫原始碼檔案 用途
src/lib 包含你的庫專案的邏輯和數據。與應用專案類似，庫專案可以包含元件、服務、模組、指令和通道。
src/public-api.ts 指定從你的函式庫匯出的所有檔案。
ng-package.json 由 ng-packagr 用於建立你的函式庫的設定檔案。
package.json 設定此函式庫所需的 npm 套件依賴。
tsconfig.lib.json 函式庫特定的 TypeScript 設定，包括 Angular 編譯器選項。
tsconfig.lib.prod.json 函式庫特定的 TypeScript 設定，在生產模式下建立庫時使用。
tsconfig.spec.json 用於函式庫的單元測試的 TypeScript 設定。
