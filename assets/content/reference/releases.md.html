Angular 的版本与发布

edit

你肯定希望 Angular 框架具有稳定性（stability）。稳定性可以确保组件与库、教程、工具和现有实践不会突然被弃用。稳定性是让基于 Angular 的生态系统变得繁荣的基石。

我们也和你一样希望 Angular 能持续演进。我们会努力确保这些你用于构建应用的基础能得到持续的改进，并让你能及时同步到 Web 生态系统的其它部分的最新进展，用户需求也是一样。

本文档包含一些我们所遵循的实践，它让我们能为你提供一个前沿的应用开发平台，同时兼顾稳定性。我们会努力确保将来的变化总是以一种可预期的方式引入。我们希望每个 Angular 用户都明白我们将在何时添加以及如何添加新特性，并且为那些将要移除的、准备弃用的特性提前做好准备。

有时，诸如移除 API 或特性等 重大变更是必要的，以便创新并跟上不断发展的最佳实践、变化的依赖项或 Web 平台的变迁。这些重大变更会经过弃用流程，详见我们的 弃用策略。

为了使这些过渡尽可能简单，Angular 团队做出了以下承诺：

  • 我们会尽量减少重大变更的数量，并尽可能提供迁移工具。
  • 我们会遵循这里所讲的弃用策略，让你有时间把应用升级到最新的 API 和最佳实践。

有帮助的：本文档中描述的实践适用于 Angular 2.0 及更高版本。如果你当前使用的是 AngularJS，请参阅 从 AngularJS 升级。 AngularJS 是所有 v1.x 版本的名称。

Angular 版本控制

Angular 版本号表示发布引入的变更级别。这种使用 语义化版本控制有助于你了解升级到新版本的潜在影响。

Angular 的版本号包括三个部分：major.minor.patch。比如，版本 7.2.11 表示主版本号是 7，小版本号是 2，补丁版本号是 11。

版本号是根据本次发布中包含的变更的级别进行递增的。

变更级别 详情
主要版本发布 包含重要的新特性，更新过程中预计需要一些但很少的开发者协助。在更新到新主要版本时，你可能需要运行更新脚本、重构代码、运行额外测试并学习新 API。
次要版本发布 包含某些新的小特性。次要版本发布完全向后兼容；在更新时不需要开发者协助，但你可以选择性地修改你的应用和库，以开始使用版本中新增的 API、特性和功能。我们在次要版本中通过扩展支持的版本来更新对等依赖项，但不要求项目更新这些依赖项。
补丁版本发布 低风险的 bug 修复版本发布。在更新时不需要开发者协助。

有帮助的提示：从 Angular 版本 7 开始，Angular 核心和 CLI 的主要版本是对齐的。这意味着为了在开发 Angular 应用时使用 CLI， @angular/core 和 CLI 的版本需要保持一致。

预览版本

我们通过提供每个主要和次要版本的 "Next" 和候选发布版本 ( rc) 预发布来让你预览即将到来的内容：

预发布类型 详情
下一个 正在积极开发和测试的版本。Next 版本以附加 -next 标识的版本标签表示，例如 8.1.0-next.0。
候选发布 特性完整并在最终测试中的版本。候选发布以附加 -rc 标识的版本标签表示，例如版本 8.1.0-rc.0。

最新的 next 或 rc 预发布版本的文档可在 next.angular.dev 获取。

发布频率

我们会定期发布新版本，以便随着 Angular 的不断演进，你可以提前计划并协调这些升级工作。

有帮助的提示：日期仅作为一般指导，可能会有所变动。

通常的发布周期如下：

  • 每 6 个月一个主版本
  • 每个主版本中包含 1~3 个小版本
  • 差不多每周一个发行版或预发行版（next 或 rc）的补丁版本

这种发布的节奏能让渴望新功能的开发者在这些功能开发开发完成并通过我们的代码审查和集成测试流程后立即就可以使用，同时为那些喜欢在新功能经过 Google 和其他使用预发布版本的开发人员的验证后才采纳的生产环境用户，保持平台的稳定性和可靠性。

LTS 修复

作为一个通用的规则，如果解决了下列问题之一，就会考虑对 LTS 版本进行修复：

  • 一个新发现的安全漏洞。
  • LTS 发布以后，由于第三方更改引起的回归性问题，比如浏览器的新版本。

弃用政策

当 Angular 团队计划移除某个 API 或特性时，会将其标记为 已弃用。这是因为某个 API 已过时，被另一个 API 取代，或因其他原因停用。已弃用的 API 会在其弃用阶段继续保留，最短持续两个主要版本（大约一年）。

为了保证你能有充足的时间和清晰的路径进行升级，我们制定了如下弃用策略：

弃用阶段 详情
公告 我们在 变更日志中宣布弃用的 API 和特性。弃用的 API 会在 文档中以 删除线显示。当我们宣布弃用时，我们还会宣布一个推荐的更新路径。此外，所有弃用的 API 都会在相应的文档中添加 @deprecated注解，这样如果你的项目依赖于这些 API，文本编辑器和 IDE 就能提供提示。
弃用周期 当某个 API 或特性被弃用时，它至少会在接下来的两个主要版本中（至少 12 个月内）保留。在此之后，弃用的 API 和特性会被列为移除候选对象。弃用可以在任何版本中宣布，但弃用的 API 或特性只有在主要版本中才会移除。在弃用的 API 或特性被移除之前，它会根据 LTS 支持策略进行维护，这意味着只有关键和安全问题会被修复。
npm 依赖项 我们只在主要版本中进行需要对你的应用进行更改的 npm 依赖更新。在次要版本中，我们通过扩展支持的版本来更新 peer 依赖，但我们不会要求项目在未来主要版本之前更新这些依赖。这意味着在次要 Angular 版本发布期间，Angular 应用和库中的 npm 依赖更新是可选的。

兼容性政策

Angular 是许多包、子项目和工具的集合。为了防止意外使用私有 API，并让你能清楚地了解哪些内容包含在这里描述的实践中 —— 我们记录了哪些内容被视为我们的公共 API 表面。详情请参见 Angular 的支持公共 API 表面。

为了保证 Angular 的向后兼容性，我们在合并任何更改之前进行一系列检查：

  • 单元测试和集成测试
  • 比较更改前后的公共 API 表面的类型定义
  • 运行所有依赖 Angular 的 Google 应用的测试

对公共 API 表面的任何更改都按照之前描述的版本控制、支持和弃用政策进行。在特殊情况下，如关键安全补丁，修复可能会引入向后不兼容的更改。此类特殊情况会通过框架的官方沟通渠道发布明确通知。

重大更改政策和更新路径

重大更改要求你进行调整，因为更改后的状态与更改前的状态不兼容。你可以在 兼容性政策中找到此规则的罕见例外。重大更改的例子包括移除公共 API 或更改 Angular 的类型定义、更改调用的时间点，或更新到包含重大更改的 Angular 依赖的新版本。

为了在 Angular 中支持你应对重大更改：

  • 我们在移除公共 API 之前遵循我们的 弃用政策
  • 通过 ng update命令支持更新自动化。它提供代码转换，我们通常会在 Google 的数十万个项目中提前进行测试
  • 逐步说明如何从一个主要版本更新到另一个版本，请参见 "Angular 更新指南"

你可以 ng update 到任何版本的 Angular，前提是满足以下条件：

  • 你要更新到的版本是受支持的。
  • 你想要更新的版本 从你想要升级到的版本中不超过一个主要版本。

比如，你可以从版本 11 更新到版本 12，前提是版本 12 仍受支持。如果要跨多个主要版本进行更新，请每次更新一个主要版本。比如，要从版本 10 更新到版本 12 时：

  1. 从版本 10 更新到版本 11。
  2. 从版本 11 更新到版本 12。

开发者预览

有时我们会在“开发者预览”标签下介绍新的 API。这些是功能齐全且经过优化的 API，但我们还没有准备好根据正常的弃用政策来稳定它们。

这可能是因为我们希望在稳定之前从真实应用程序收集反馈，或者因为相关的文档或迁移工具不完全完整。

本文档中描述的政策和实践不适用于标记为 Developer Preview 的 API。此类 API 可以随时更改，即使在框架的新补丁版本中也是如此。团队应该自己决定使用开发者预览版 API 的好处是否值得冒险破坏我们正常使用语义版本控制之外的更改。

实验性

这些 API 可能永远不会变得稳定，或者在变得稳定之前会有重大变化。

本文档中描述的政策和做法不适用于标记为实验性的 API。此类 API 可能随时更改，即使在框架的新补丁版本中也是如此。团队应自行决定使用实验性 API 的好处是否值得在我们通常使用的语义化版本控制之外承受破坏性更改的风险。
