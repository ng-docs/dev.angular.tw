Angular 版本控制與發布

edit

我們認識到你需要 Angular 框架的穩定性。穩定性確保了可複用的元件和函式庫、教程、工具以及學到的實踐經驗不會意外地過時。穩定性對於 Angular 周圍生態系統的蓬勃發展至關重要。

我們也和你一樣，認為 Angular 需要不斷發展。我們努力確保你正在建立的基礎設施能夠持續改進，使你能夠與 Web 生態系統的其他部分以及使用者的需求保持同步。

本文件包含我們遵循的實踐，旨在為你提供一個領先的應用開發平台，並在穩定性方面取得平衡。我們力求確保未來的變更始終以可預測的方式引入。我們希望每個依賴 Angular 的人都知道何時以及如何新增新屬性，並在過時的屬性被刪除時做好充分準備。

有時，為了創新並與不斷發展的最佳實踐、不斷變化的依賴項或 Web 平台的轉變保持同步，破壞性變更（例如刪除 API 或屬性）是必要的。這些破壞性變更會經歷一個棄用過程，這在我們的棄用策略中進行瞭解釋。

為了使這些轉場儘可能簡單明瞭，Angular 團隊做出了以下承諾：

  • 我們努力減少破壞性變更的數量，並在可能的情況下提供遷移工具
  • 我們遵循此處描述的棄用策略，以便你有時間將你的應用更新到最新的 API 和最佳實踐

提示： 本文件中描述的實踐適用於 Angular 2.0 及更高版本。如果你目前正在使用 AngularJS，請參閱從 AngularJS 升級。AngularJS 是所有 v1.x 版本 Angular 的名稱。

Angular 版本控制

Angular 版本號指示了版本發布所引入的變更級別。這種語義化版本控制的使用有助於你瞭解更新到新版本的潛在影響。

Angular 版本號有三個部分：major.minor.patch。例如，版本 7.2.11 表示主版本 7，次版本 2 和補丁級別 11。

版本號會根據版本發布中包含的變更級別進行遞增。

變更級別 詳情
主版本發布 包含重要的新屬性，更新期間預計需要少量但最少的開發者協助。當更新到新的主版本發布時，你可能需要執行更新指令碼、重構程式碼、執行額外的測試以及學習新的 API。
次版本發布 包含新的較小屬性。次版本發布是完全向後相容的；更新期間預計不需要開發者協助，但是你可以選擇修改你的應用和函式庫，開始使用版本發布中新增的新 API、屬性和功能。我們在次版本中透過擴充套件支援的版本來更新對等依賴，但是我們不要求專案更新這些依賴。
補丁版本發布 低風險的缺陷修復版本發布。更新期間預計不需要開發者協助。

提示： 從 Angular 版本 7 開始，Angular core 和 CLI 的主版本已對齊。這意味著為了在你開發 Angular 應用時使用 CLI，@angular/core 和 CLI 的版本需要相同。

預覽版本發布

我們透過為每個主版本和次版本發布提供 "Next" 和發布候選版本 (rc) 預覽版本本，讓你預覽即將發布的內容：

預覽版本本型別 詳情
Next 正在積極開發和測試的版本。next 版本由附加了 -next 識別符號的版本標記指示，例如 8.1.0-next.0。
發布候選版本 一個屬性完整且正在進行最終測試的版本。發布候選版本由附加了 -rc 識別符號的版本標記指示，例如版本 8.1.0-rc.0。

最新的 next 或 rc 預覽版本本的文件可在 next.angular.dev 上找到。

發布頻率

我們致力於定期的版本發布計劃，以便你可以計劃和協調你的更新，使其與 Angular 的持續發展保持同步。

提示： 日期僅作為一般指導，可能會發生變化。

通常，可以預期以下發布週期：

  • 每 6 個月一次主版本發布
  • 每個主版本發布套件含 1-3 個次版本發布
  • 幾乎每週都會發布補丁版本和預覽版本本（next 或 rc）建立

這種發布節奏使渴望嚐鮮的開發者能夠在其新屬性完全開發並透過我們的程式碼審查和整合測試流程後立即訪問這些屬性，同時保持平台的穩定性和可靠性，以滿足喜歡在屬性經過 Google 和其他使用預覽版本本建立的開發者驗證後再接收這些屬性的生產使用者的需求。

LTS 修復

作為一般規則，如果修復是為瞭解決以下問題之一，則 LTS 版本會考慮該修復：

  • 新發現的安全漏洞，
  • 自 LTS 開始以來，由第三方更改（例如新的瀏覽器版本）引起的迴歸。

棄用策略

當 Angular 團隊打算移除某個 API 或屬性時，它將被標記為 deprecated（已棄用）。當某個 API 過時、被另一個 API 取代或以其他方式停止使用時，就會發生這種情況。已棄用的 API 在其棄用階段仍然可用，該階段至少持續兩個主要版本（大約一年）。

為了幫助確保您有足夠的時間和清晰的更新路徑，以下是我們的棄用策略：

棄用階段 詳情
公告 我們在 變更日誌 中宣佈已棄用的 API 和屬性。已棄用的 API 會在 文件 中以 刪除線 的形式出現。當我們宣佈棄用時，我們還會宣佈推薦的更新路徑。此外，所有已棄用的 API 都在相應的文件中使用 @deprecated 進行註解，這使文字編輯器和 IDE 能夠在您的專案依賴於它們時提供提示。
棄用週期 當 API 或屬性被棄用時，它至少在接下來的兩個主要版本（至少 12 個月的時間段）中仍然存在。在此之後，已棄用的 API 和屬性將成為移除的候選物件。可以在任何版本中宣佈棄用，但已棄用 API 或屬性的移除僅在主要版本中發生。在已棄用的 API 或屬性被移除之前，它會根據 LTS 支援策略進行維護，這意味著只會修復關鍵問題和安全問題。
npm 依賴 我們只在主要版本中進行需要更改您的應用的 npm 依賴更新。在次要版本中，我們透過擴充套件支援的版本來更新對等依賴，但我們不要求專案在未來的主要版本之前更新這些依賴。這意味著在 Angular 的次要版本發布期間，Angular 應用和庫中的 npm 依賴更新是可選的。

相容性策略

Angular 是許多軟體套件、子專案和工具的集合。 為了防止意外使用私有 API，並使您可以清楚地瞭解此處描述的實踐涵蓋的內容，我們記錄了哪些被認為是我們的公共 API 表面，哪些不是。 有關詳細訊息，請參閱 Angular 支援的公共 API 表面。

為了保證 Angular 的向後相容性，我們在合併任何更改之前都會執行一系列檢查：

  • 單元測試和整合測試
  • 比較更改前後公共 API 表面的型別定義
  • 執行 Google 中所有依賴 Angular 的應用的測試

對公共 API 表面的任何更改均根據先前描述的版本控制、支援和棄用策略進行。在特殊情況下，例如關鍵安全補丁，修復程式可能會引入向後不相容的更改。此類特殊情況會附帶框架官方溝通渠道上的明確通知。

破壞性變更策略和更新路徑

破壞性變更需要您進行一些工作，因為變更後的狀態與變更前的狀態不向後相容。您可以在 相容性策略 中找到此規則的罕見例外情況。破壞性變更的範例包括刪除公共 API 或 Angular 型別定義的其他更改、更改呼叫的時序或更新到 Angular 依賴項的新版本，而新版本本身包含破壞性變更。

為了在 Angular 中出現破壞性變更時為您提供支援：

  • 在移除公共 API 之前，我們遵循 棄用策略
  • 透過 ng update 指令支援更新自動化。它提供了程式碼轉換，我們通常在 Google 的數十萬個專案中提前測試過這些轉換
  • 在 『Angular 更新指南』 中，提供了如何從一個主要版本更新到另一個主要版本的逐步說明

您可以 ng update 到任何 Angular 版本，前提是滿足以下條件：

  • 您要更新到的版本受支援。
  • 您要更新自的版本在您要升級到的版本的一個主要版本之內。

例如，您可以從版本 11 更新到版本 12，前提是版本 12 仍然受支援。 如果您想跨多個主要版本更新，請一次執行一個主要版本的更新。 例如，要從版本 10 更新到版本 12：

  1. 從版本 10 更新到版本 11。
  2. 從版本 11 更新到版本 12。

開發者預覽版

有時我們會以『開發者預覽版』的標籤引入新的 API。這些 API 功能齊全且經過完善，但我們尚未準備好根據我們正常的棄用策略對其進行穩定。

這可能是因為我們希望在穩定之前收集來自真實應用的反饋，或者因為相關的文件或遷移工具尚未完全完成。可以透過 GitHub issue 提供反饋，開發者可以在其中分享他們的經驗、報告錯誤或提出改進建議，以幫助改進該屬性。

本文件中描述的策略和實踐不適用於標記為『開發者預覽版』的 API。此類 API 可能會隨時更改，即使在框架的新補丁版本中也是如此。團隊應自行決定，使用『開發者預覽版』API 的好處是否值得冒語義化版本控制正常使用之外的破壞性變更的風險。

實驗性的

這些 API 可能根本不會變得穩定，或者在變得穩定之前會發生重大變化。

本文件中描述的策略和實踐不適用於標記為實驗性的 API。此類 API 可能會隨時更改，即使在框架的新補丁版本中也是如此。團隊應自行決定，使用實驗性 API 的好處是否值得冒語義化版本控制正常使用之外的破壞性變更的風險。
