註冊回呼函式，使其在應用下次完成渲染時，在指定的階段期間被呼叫。可用的階段有：

  • earlyRead 使用此階段在後續的 write 回呼之前從 DOM 中讀取，例如執行瀏覽器本身不支援的自定義佈局。如果讀取可以等到寫入階段之後，請優先使用 read 階段。絕不在此階段寫入 DOM。
  • write 使用此階段寫入 DOM。絕不在此階段從 DOM 中讀取。
  • mixedReadWrite 使用此階段同時讀取和寫入 DOM。如果可以將工作分配到其他階段，絕不要使用此階段。
  • read 使用此階段從 DOM 中讀取。絕不在此階段寫入 DOM。

如果可能，您應該優先使用 read 和 write 階段，而不是 earlyRead 和 mixedReadWrite 階段，以避免效能下降。

請注意：

  • 回呼函式以下列階段順序在下次渲染之後執行一次：
    1. earlyRead
    2. write
    3. mixedReadWrite
    4. read
  • 同一階段中的回呼函式按照它們註冊的順序執行。
  • 回呼函式僅在瀏覽器平台上執行，它們不會在服務器上執行。

作為此規範一部分執行的第一個階段回呼將不接收任何引數。此規範中每個後續階段的回呼都將接收先前執行的階段回呼的回傳值作為引數。這可以用於協調跨多個階段的工作。

Angular 無法驗證或強制階段是否被正確使用，而是依賴於每位開發者遵循為每個值記錄的指南，並仔細選擇合適的階段，並在必要時重構他們的程式碼。透過這樣做，Angular 能夠更好地最大限度地減少與手動 DOM 訪問相關的效能下降，從而確保為你的應用或函式庫的最終使用者提供最佳體驗。

不能保證元件在回呼執行之前完成水合。 直接讀取或寫入 DOM 和佈局時，您必須謹慎。

註冊一個回呼函式，使其在應用下次完成渲染時，在 mixedReadWrite 階段期間被呼叫。

您應該優先為回呼指定一個顯式的階段，否則您將面臨嚴重的效能下降風險。

請注意，回呼函式將執行

  • 按照其註冊的順序
  • 僅在瀏覽器平台上
  • 在 mixedReadWrite 階段期間
